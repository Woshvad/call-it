Root causes (from 08-UAT.md):
1. NO Farcaster Mini App SDK anywhere — grep for `@farcaster/miniapp-sdk` / `sdk.actions.ready()` returns nothing (only `@farcaster/auth-kit` from Phase 1.5). Farcaster Mini Apps stay on a blank/splash screen until the app calls `sdk.actions.ready()`; it is never called.
2. The `/call/[id]` page is 100% client-rendered (SSR emits BAILOUT_TO_CLIENT_SIDE_RENDERING — empty server shell) behind a heavy Privy+wagmi stack that does not initialize/render in the restricted Mini App webview.

Scope (MINIMUM — render only, per D-01 / Phase-10 boundary):
- Add `@farcaster/miniapp-sdk` (verified current package, v0.3.0 — see package legitimacy gate below) and call `sdk.actions.ready()` once the receipt content is mounted.
- Ensure the READ-ONLY receipt renders in the Mini App webview WITHOUT blocking on Privy/wagmi wallet init — a logged-out webview must still show the public receipt.

Purpose: Make the launch TARGET an actual Mini App that reveals the receipt, closing the loop opened in 08-02 (manifest + `fc:miniapp` launch_miniapp action + launch button) which shipped the launch button but never made the destination a real Mini App.

Output:

- A CI test asserting ready() is invoked once and is fail-safe when the SDK/host is absent.

<approach_note>
**Why a separate `MiniAppReady` client component, not a page edit only.** `sdk.actions.ready()` must run client-side after the receipt DOM is mounted; isolating it in a tiny `'use client'` component (a) keeps the SDK import out of the heavy page module's top-level eval, (b) lets it be mounted on every status branch of the page (Live / Settled / CallerExited / loading), and (c) makes it independently unit-testable.

**Decouple the read-only receipt from wallet init.** The receipt's public content (handle, market line, outcome, stats) comes from `fetchCallData` (relayer `/live-state`) and the subgraph — NOT from the connected wallet. The wallet/Privy stack is only needed for the INTERACTIVE controls (Follow/Fade/Challenge/exit), which are explicitly Phase-10/out-of-scope for the Mini App. So the page must render its read-only receipt regardless of Privy/wagmi readiness:
- The loading gate (`if (isLoadingCall && !callData)`, page.tsx:1325) keys off the relayer fetch, NOT wallet readiness — confirm it does not block on `authenticated`/`useAccount`.
- `MiniAppReady` mounts immediately (not gated on wallet) and calls `ready()` once `callData` is present (or once the loading skeleton has rendered), so the host dismisses the splash even for a logged-out viewer.

CLAUDE.md constraints: no new contract/USDC addresses; no mainnet 42161 transact path (D-01); read-only render only.
</approach_note>

       - Use a `useRef` guard so `ready()` is called at most once even under StrictMode double-invoke.
       - Accept an optional `enabled?: boolean` prop (default true) so the page can defer ready() until `callData` is present (so the host reveals content, not an empty skeleton). When `enabled` is false, do nothing yet.

       - asserts `signalReady` invokes the injected mock `ready` exactly once,
