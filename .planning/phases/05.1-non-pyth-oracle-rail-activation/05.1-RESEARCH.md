## Summary

Phase 05.1 closes the Acknowledged Gap B from Phase 4: making all 6 non-Pyth settlement paths (NFT-TWAP, DefiLlama, RpcMetrics, Snapshot, Tally, CEX) settle end-to-end on Arbitrum Sepolia. The security-critical on-chain attestation rail (`submitAttestation` + EIP-712 + `ECDSA.recover` + per-type signer registry) is **already implemented and 12/12 tests GREEN**. What blocks end-to-end settlement is three functional gaps:

**Gap B.1 — Empty adapterMap:** `setAdapterMap` exists at `SettlementManager.sol:622` but `DeployPhase4.s.sol` never calls it. The live SM has `adapterMap[*][*] == OracleAdapter.Pyth(0)` for every slot, so every non-Pyth `submitAttestation` call reverts via `_checkAdapterBinding` (`:592`) because `oracleType != 0`. Fix: a new deploy script calls `setAdapterMap` for every configured `(marketType, eventSubtype)` pair — see Standard Stack section for the exact pairs.

**Gap B.2 — Legacy attestation format in 6 adapters + settlement-watcher:** All 6 adapters (`snapshot-adapter.ts`, `tally-adapter.ts`, `defillama-adapter.ts`, `rpc-metrics-adapter.ts`, `nft-twap-adapter.ts`, `cex-adapter.ts`) and `settlement-watcher.ts` use per-adapter EIP-712 domains (`CallIt-SnapshotTally`, `CallIt-DefiLlama`, …) that DO NOT match the deployed contract's unified `CallIt-Oracle` domain. Every attestation produced by these adapters will fail `ECDSA.recover` on-chain. All 6 must be rewired onto `oracle-attestation.ts::signOracleAttestation`. `settlement-watcher.ts` dispatches on `marketType` directly (line 422) instead of reading `adapterMap` from the chain — it must dispatch on the on-chain adapterMap result.

**Gap B.3 — No off-chain criteria store:** The on-chain `Call` struct does not carry string identifiers (`proposalId`, `protocolSlug`, `tokenSymbol`). These are needed by the adapters at settlement time. There is currently NO relayer Postgres table or subgraph field that stores them. The settlement-watcher comments acknowledge this (lines 420-421, 582, 616, 649-650) and use `call.assetA_feedId` as a raw proxy. The plan must decide the retrieval strategy and implement it.

**The central schema problem — SETTLE-DUAL-GOV:** `EventSubtype.Governance = 6` is a single enum value (`ICallRegistry.sol:30`), but `OracleAdapter.Snapshot = 4` and `OracleAdapter.Tally = 5` are two distinct adapters. `adapterMap[marketType][6]` can hold exactly one adapter, creating a collision. The recommended resolution is **Option A — split EventSubtype**: add `Governance_Snapshot = 6` and `Governance_Tally = 7`, shifting `ProtocolMilestone` to `8`. SettlementManager is non-upgradeable and must be redeployed regardless (for B.1 deploy script fix); since CallRegistry uses the ICallRegistry enum, adding two values to that interface requires a CallRegistry source update + redeploy. The FULL ripple is precisely scoped below.

**Primary recommendation:** Option A (enum split) + a single `DeployPhase5_1.s.sol` that redeploys both contracts, wires all 4 downstream contracts, calls `setAdapterMap` for all 8 configured pairs, calls `setAttestationSigner` for all 6 non-Pyth oracle types, updates `addresses.ts`, redeploys the subgraph, and re-runs the Phase 4 money-path tests against the new addresses.

## The Central Design Problem: SETTLE-DUAL-GOV

### Current State (verified in source)

**`ICallRegistry.sol:23-32`** [VERIFIED: source at file:line]
```solidity
enum EventSubtype {
    None,              // 0
    TvlMilestone,      // 1
    VolumeFees,        // 2
    OnchainMetric,     // 3
    CexListing,        // 4
    TokenLaunch,       // 5
    Governance,        // 6  <-- SINGLE VALUE
    ProtocolMilestone  // 7
}
```

**`ISettlementManager.sol:24-31`** [VERIFIED: source at file:line]
```solidity
enum OracleAdapter {
    Pyth = 0,
    NftTwap = 1,
    DefiLlama = 2,
    RpcMetrics = 3,
    Snapshot = 4,      <-- SNAPSHOT
    Tally = 5,         <-- TALLY
    CexScraper = 6
}
```

**`SettlementManager.sol:119-120`** [VERIFIED: source at file:line]
```solidity
mapping(uint8 => mapping(uint8 => OracleAdapter)) public adapterMap;
// adapterMap[marketType][eventSubtype] -> OracleAdapter
// COLLISION: adapterMap[2][6] can only hold ONE of {Snapshot(4), Tally(5)}
```

**`SettlementManager.sol:589-593`** [VERIFIED: source at file:line] — `_checkAdapterBinding`:
```solidity
function _checkAdapterBinding(uint256 callId, uint8 oracleType) internal view {
    ICallRegistry.Call memory call = callRegistry.getCall(callId);
    OracleAdapter configuredAdapter = adapterMap[uint8(call.marketType)][uint8(call.eventSubtype)];
    if (oracleType != uint8(configuredAdapter)) revert InvalidAttestation();
}
```

The collision: if `adapterMap[2][6] = Snapshot(4)`, then a Tally attestation (`oracleType=5`) is rejected by `_checkAdapterBinding`. If set to `Tally(5)`, Snapshot attestations are rejected. There is no way to support both Snapshot and Tally for `eventSubtype=6` without changing either the enum or the map structure.

### Option A: Split EventSubtype (RECOMMENDED)

**What:** Add `Governance_Snapshot = 6` and `Governance_Tally = 7`, shifting `ProtocolMilestone` to `8`.

**ICallRegistry.sol change:**
```solidity
enum EventSubtype {
    None,                 // 0
    TvlMilestone,         // 1
    VolumeFees,           // 2
    OnchainMetric,        // 3
    CexListing,           // 4
    TokenLaunch,          // 5
    Governance_Snapshot,  // 6  — NEW
    Governance_Tally,     // 7  — NEW
    ProtocolMilestone     // 8  — renumbered from 7
}
```

**Contract diff surface:**
- `ICallRegistry.sol` — enum body only (enum comment + two new values + renumber ProtocolMilestone)
- `CallRegistry.sol` — NO code change (uses the interface enum, no explicit integer comparisons in source found)
- `SettlementManager.sol` — NO code change (dispatch reads adapterMap; enum values are ABI encoded)
- `DeployPhase5_1.s.sol` — set `adapterMap[2][6] = Snapshot(4)`, `adapterMap[2][7] = Tally(5)`, `adapterMap[2][8] = DefiLlama(2)` for ProtocolMilestone (or another adapter per spec)

**Storage-layout impact:** Neither CallRegistry nor SettlementManager has enum values stored as integers in their slots that would be affected by renumbering ProtocolMilestone. The only stored `eventSubtype` values are in `Call` structs stored in `_calls[callId]`. If any live calls have `eventSubtype = 7 (ProtocolMilestone)`, they will be mis-decoded as `Governance_Tally` after the source update. On Sepolia staging with seeded test calls this matters; verify no live calls use subtype 7 before redeploy. On fresh redeploy from a new block there is no issue.

**Ripple:**
1. `CallRegistry.sol` source + redeploy (Sepolia: new address; record in `addresses.ts`)
2. `packages/shared/src/constants/addresses.ts` — `CALL_REGISTRY_ARBITRUM_SEPOLIA` updated
3. `SettlementManager.sol` redeploy (Sepolia: new address because it's non-upgradeable) — `SETTLEMENT_MANAGER_ARBITRUM_SEPOLIA` updated
4. All 4 downstream `setSettlementManager` calls + `setAuthorizedRepWriter` + `setStylusScoreEngine`
5. `setAdapterMap` called for all 8 pairs (see table below)
6. `setAttestationSigner` called for all 6 non-Pyth oracle types
7. Subgraph `subgraph.yaml`: new CR + SM addresses + startBlocks
8. Subgraph `schema.graphql`: `eventSubtype: Int` field unchanged (still Int); no schema migration needed — the numeric values are what's stored
9. Subgraph handler `call-registry.ts`: `call.eventSubtype = 0` default unchanged; no handler reads specific eventSubtype values currently
10. Frontend `new/` call form: `EVENT_SUBTYPE_TO_UINT` map in `@call-it/shared` must add `governance_snapshot: 6`, `governance_tally: 7` and adjust `protocolMilestone: 8`
11. Frontend governance call creation form (`EventFields.tsx`): change `'governance'` option to let user choose between Snapshot and Tally

**oracleType binding preserved:** With Option A, each governance source gets its own `(marketType, eventSubtype)` slot, so `_checkAdapterBinding` will find the correct configured adapter for each.

**Complete adapterMap after Option A** (8 pairs to set):

| marketType | eventSubtype | adapterMap value |
|---|---|---|
| `Event(2)` | `TvlMilestone(1)` | `DefiLlama(2)` |
| `Event(2)` | `VolumeFees(2)` | `DefiLlama(2)` |
| `Event(2)` | `OnchainMetric(3)` | `RpcMetrics(3)` |
| `Event(2)` | `CexListing(4)` | `CexScraper(6)` |
| `Event(2)` | `TokenLaunch(5)` | `CexScraper(6)` |
| `Event(2)` | `Governance_Snapshot(6)` | `Snapshot(4)` |
| `Event(2)` | `Governance_Tally(7)` | `Tally(5)` |
| `Event(2)` | `ProtocolMilestone(8)` | `DefiLlama(2)` or `RpcMetrics(3)` — per spec §4.3.7 |
| `PriceTarget(0)` | `None(0)` | `Pyth(0)` |
| `SpreadVs(1)` | `None(0)` | `Pyth(0)` |
| `Event(2)` | `None(0)` | `Pyth(0)` (fallback; not used for seeded calls) |

Note: NFT calls use `PriceTarget(0)` + `None(0)` with `assetA = address(NFT)` — the adapter decision is actually made by the relayer based on whether assetA is a Pyth feedId or an NFT address. The on-chain adapterMap maps `PriceTarget` to `Pyth(0)`, and the relayer uses the on-chain adapterMap query to select the adapter. For the NFT-TWAP path specifically, the relayer must call `adapterMap(0, 0)` and get `Pyth(0)`, but must then check if the call is an NFT call (assetA is a contract address, not a Pyth feed ID). This pre-existing NFT routing logic is already in the watcher (line 487-515); just needs to use `signOracleAttestation` instead of the legacy format.

### Option B: Third dimension / composite key

**What:** Change `adapterMap` to `mapping(uint8 => mapping(uint8 => mapping(uint8 => OracleAdapter)))` or use a composite `(marketType << 8 | eventSubtype)` key.

**Why not recommended:**
- Requires SettlementManager source change (new mapping declaration)
- Does NOT eliminate the need to redeploy SM (it's non-upgradeable anyway)
- More complex dispatch in `_dispatchOracle` and `_checkAdapterBinding`
- The oracleType binding is preserved more cleanly with Option A's unique (mT, eS) pairs
- No benefit over Option A given that SM must be redeployed regardless

### Option C: Any other design

**What:** e.g., store oracleType directly in the Call struct at creation time.

**Why not recommended:** Requires CallRegistry source change + ICallRegistry interface change (bigger diff), and moves the oracle-type decision to call creation time (which creates future coupling if adapters change).

## Gap B.1: setAdapterMap Deploy Script

### Current State [VERIFIED: `packages/contracts/script/DeployPhase4.s.sol`]

`DeployPhase4.s.sol` deploys SM, wires `setSettlementManager` on 4 contracts, calls `setAuthorizedRepWriter`, funds ETH. It does NOT call `setAdapterMap` or `setAttestationSigner`. The live SM has `adapterMap[*][*] = OracleAdapter(0) = Pyth` for every slot (Solidity mapping default = 0).

Every non-Pyth `submitAttestation` call REVERTS. Safe-closed, but non-functional.

## Gap B.3: Off-Chain Call-Criteria Retrieval

### The Problem [VERIFIED: no criteria store found in relayer DB schema]

The on-chain `Call` struct (verified: `ICallRegistry.sol:130-157`) stores:
- `assetA: uint256` — Pyth feed key as `uint256(bytes32(feedId))` for price calls, or `uint256(address)` for NFT/governance
- `criteriaHash: bytes32` — `keccak256(criteriaText)` — HASH ONLY, not the original string
- `targetValue: uint256` — numeric target

It does NOT store: `proposalId` (string), `protocolSlug` (string), `tokenSymbol` (string).

For governance calls specifically, there is no standard way to encode a Snapshot/Tally `proposalId` (a hexadecimal string like `0xabcd...`) into a `uint256 assetA`. CEX calls need the `tokenSymbol` string.

The settlement-watcher currently uses the raw `call.assetA_feedId` (wrong ABI name) as a placeholder for these identifiers (lines 523, 583, 616, 650). This is clearly acknowledged as a stub.

### Retrieval Strategy Options

**Option R1 — Criteria store in relayer Postgres (RECOMMENDED for Phase 05.1)**

At call creation time, the relayer's `calls-preflight.ts` already receives `criteriaText` from the frontend (`calls-preflight.ts:61`). Add a new Postgres table `call_oracle_criteria`:

```sql
CREATE TABLE call_oracle_criteria (
  call_id     BIGINT PRIMARY KEY,  -- on-chain callId (assigned post-tx)
  oracle_type SMALLINT NOT NULL,   -- OracleType enum value
  identifier  TEXT NOT NULL,       -- proposalId | protocolSlug | tokenSymbol
  target_unit TEXT,                -- unit description for targetValue comparison
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

**Insertion point:** After `CallCreated` event is indexed by the relayer event watcher, the relayer maps `criteriaText` → parse out the identifier → insert. Alternatively, the frontend could POST this data explicitly.

**Risk:** Criteria store availability. If the store is missing for a callId, settlement-watcher must DEFER (return ambiguous), not mis-settle.

**Option R2 — Read from the subgraph's `criteriaText` field**

The subgraph currently does NOT index `criteriaText` (the string is not in the `CallCreated` event — only `criteriaHash` is emitted). The subgraph `Call` entity has no `criteriaText` field (`schema.graphql:12-31`). Adding it would require: (a) emitting `criteriaText` in the event (CallRegistry source change, which Phase 05.1 is already doing for Option A), or (b) off-chain enrichment.

**Option R3 — Parse criteriaText at settlement from criteriaHash**

Impossible — `keccak256` is one-way. The string cannot be recovered from the hash.

**Option R4 — Store identifier in assetA at call creation**

For governance calls, store `uint256(bytes32(proposalId))` in `assetA`. This is what the watcher currently assumes. The problem: Snapshot proposalIds are 32-byte hashes that fit in `bytes32` / `uint256`. Tally proposalIds are numeric (on-chain `proposalId` from Governor contracts) which also fit in `uint256`. Protocol slugs (DefiLlama) are human-readable strings like `"uniswap"` that do NOT fit in `uint256`. Token symbols (CEX) are short strings but also not `uint256`. This pattern only works for governance calls.

**Recommended approach for Phase 05.1:**
- **Governance (Snapshot/Tally):** Use `assetA` to store the proposalId encoded as `uint256`. For Snapshot, `uint256(bytes32(proposalId))` where proposalId is already a hex hash. For Tally, the on-chain `proposalId` is already a `uint256`. This is already how the settlement-watcher assumes it works (lines 583, 616). Validate in the call creation flow.
- **NFT-TWAP:** Use `assetA` as `uint256(address(nftContract))` — already the pattern.
- **DefiLlama / RpcMetrics / ProtocolMilestone:** Store a protocol identifier. For DefiLlama, the `protocolSlug` is a string. Use a criteria store OR store a pre-registered protocolSlug index. For MVP/Phase 05.1: add a `call_oracle_criteria` table in relayer Postgres. A minimal Drizzle migration adds this in one plan.
- **CEX listing:** tokenSymbol is a short string. Same approach: criteria store OR encode as bytes.

**Unit Verification for targetValue:**

Per the `oracle-attestation.ts` comment: "observed and target MUST be expressed in the SAME unit." The risk is that a call is created with `targetValue` in one unit (e.g., USD * 1e6) but the adapter returns a value in another unit (e.g., DefiLlama TVL in USD whole dollars). Each adapter must document its unit contract. Current adapter patterns:
- DefiLlama TVL: returns USD as a float, must be converted to match `targetValue` unit used at creation
- NFT floor: Alchemy returns ETH price; `targetValue` must be in the same unit (ETH wei or ETH * 1e6)
- Governance: binary pass/fail — `targetValue` is unused, outcome is quorum/votes-based
- RpcMetrics: depends on metric (e.g., liquidation events count vs dollar threshold)

**Fail-safe:** If criteria cannot be resolved (no store entry, no assetA match), the adapter MUST return `ambiguous = true`, which the watcher maps to a DEFER (not a settlement).

## Common Pitfalls

### Pitfall 1: Wrong chainId in EIP-712 domain (CRITICAL)

**What goes wrong:** Legacy adapters hardcode `chainId: 42161n` (`snapshot-adapter.ts:283`, `defillama-adapter.ts:263`). On Sepolia (chainId 421614), the on-chain `EIP712._domainSeparatorV4()` uses `block.chainid = 421614`. The recovered signer will not match `attestationSigner[oracleType]` → `revert InvalidAttestation()`.

**Why it happens:** The per-adapter EIP-712 domains predate the unified `oracle-attestation.ts` which correctly uses the deployment chainId.

**How to avoid:** All adapters MUST call `signOracleAttestation({ chainId: deploymentChainId, verifyingContract: smAddress, ... })`. `chainId` must come from env/config, never hardcoded.

**Warning signs:** `InvalidAttestation()` revert on `submitAttestation` calls; `oracle-attestation.test.ts` assertion "chainId is the deployment chain (421614)" failing.

### Pitfall 2: adapterMap default = 0 = Pyth

**What goes wrong:** Ethereum mapping defaults to zero. `OracleAdapter.Pyth = 0`. So `adapterMap[2][1]` (Event/TvlMilestone) returns `Pyth(0)` until explicitly set. A non-Pyth attestation for a TVL call will be rejected because `oracleType=2 != configuredAdapter=0`.

**Why it happens:** DeployPhase4.s.sol never called `setAdapterMap`.

**How to avoid:** `DeployPhase5_1.s.sol` must call `setAdapterMap` for every non-Pyth pair before returning. Post-deploy assertions: `cast call SM "adapterMap(uint8,uint8)(uint8)" 2 1` must return `2`, not `0`.

**Warning signs:** All non-Pyth `submitAttestation` calls reverting with `InvalidAttestation()` even after adapter rewiring.

### Pitfall 3: enum renumbering breaks live call data

**What goes wrong:** If any live Sepolia calls were created with `eventSubtype = 7 (ProtocolMilestone)`, and we renumber it to `8` in the new CallRegistry, those calls will be decoded as `Governance_Tally(7)` by the new contract, causing wrong adapter dispatch.

**Why it happens:** ABI-encoded enum values are stored as integers. Old `7` is now `Governance_Tally`.

**How to avoid:** Before deploying the new CallRegistry, query live calls with `eventSubtype == 7`. If any exist, migrate them (reissue as new calls with `eventSubtype=8`) or document the gap. On a fresh Sepolia staging environment (no live calls with subtype 7), this is not an issue.

**Warning signs:** Any existing call with `eventSubtype=7` routes to Tally adapter instead of ProtocolMilestone.

### Pitfall 4: oracleType binding check after partial adapterMap configuration

**What goes wrong:** If `setAdapterMap` is called for some pairs but not others during the deploy, and a settlement attempt arrives before all pairs are configured, it routes to `Pyth(0)` (the default) and the attestation fails.

**How to avoid:** All `setAdapterMap` calls MUST happen atomically in the same `vm.startBroadcast()` block. Do not configure in separate transactions.

### Pitfall 5: NftTwap uses submitAttestation but the watcher calls submitNftFloor

**What goes wrong:** `settlement-watcher.ts:508-514` calls `submitNftFloor` (a different function), while the unified rail uses `submitAttestation`. After rewiring `nft-twap-adapter.ts` to use `signOracleAttestation`, the watcher's `submitNftFloor` call must also be replaced with `SM.submitAttestation`.

**How to avoid:** Check if `submitNftFloor` is defined on ISettlementManager. If it is a separate function, the adapter rewire must change both the signing format AND the submission function call.

**Warning signs:** `submitNftFloor` call succeeding but `adapterMap[0][0]` still routing to Pyth → NFT calls don't use attestation path.

### Pitfall 6: CEX adapter does not call SM.submitAttestation

**What goes wrong:** `settlement-watcher.ts:670` — CEX case logs "attestation signed" but does NOT call `SM.submitAttestation`. The comment says "In full production, would also call SM.submitAttestation with the signed data." Without this call, `_attestationReady[callId]` is never set → CEX calls DEFER indefinitely.

**How to avoid:** After rewiring cex-adapter to `signOracleAttestation`, add the `writeContract` call to `SM.submitAttestation` in the CEX case.

### Pitfall 7: Subgraph reindex lag after contract redeploy

**What goes wrong:** After redeploying SM + CR to new addresses, the subgraph still indexes the old addresses. New events are not indexed. Frontend queries return stale data.

**How to avoid:** Update `subgraph.yaml` address + startBlock for both SM and CR, rebuild, and `graph deploy --studio`. The deployment may take 5-30 minutes to sync. Do NOT run end-to-end tests until the subgraph has caught up to the latest block.

### Pitfall 8: Settlement-watcher ENV vars not updated after redeploy

**What goes wrong:** The relayer still has the old SM address in `SETTLEMENT_MANAGER_ARBITRUM_SEPOLIA` env var. `submitAttestation` and `settle` calls go to the old (now inert) contract.

**How to avoid:** After deploy, update all relayer env files (`apps/relayer/.env`, `.env.production`) with the new SM and CR addresses. Restart the relayer worker.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---|---|---|
| V2 Authentication | no | — |
| V3 Session Management | no | — |
| V4 Access Control | yes | `onlyOwner` on `setAdapterMap`, `setAttestationSigner`; existing mitigations unchanged |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---|---|---|
| Forged attestation with wrong oracleType | Spoofing | `_checkAdapterBinding` (SettlementManager.sol:589-593); per-type `attestationSigner` registry |
| Cross-chain replay (Sepolia attestation on mainnet) | Spoofing | EIP-712 domain binds `block.chainid` + `address(this)` |
| adapterMap collision (Snapshot/Tally on same slot) | Tampering | Option A enum split gives each oracle its own (mT, eS) slot |
| Mis-settled governance call (wrong proposal outcome) | Tampering | `signOracleAttestation` enforces `outcome != Pending`; resolver is sole authority |
| Settlement without criteria resolution | Spoofing | Adapter returns `ambiguous=true` if criteria store lookup fails → DEFER, never settle |
| New enum values break existing calls | Tampering | Pre-deploy audit: verify no live calls with `eventSubtype=7` (old ProtocolMilestone) before redeploy |

### Blast-radius note

The `_checkAdapterBinding` check at `SettlementManager.sol:589-593` is the oracleType↔adapterMap binding that limits blast radius. A compromised `snapshot-tally` KMS key can only forge Snapshot(4) or Tally(5) attestations (whichever `attestationSigner` maps it to). With Option A, it cannot forge a DefiLlama(2) attestation because `oracleType=2 != attestationSigner[4]`. This control is already implemented (T-04-04-01 CLOSED).

## Open Questions (RESOLVED)

1. **ProtocolMilestone adapter assignment**
   - What we know: EventSubtype.ProtocolMilestone (will be 8 in Option A) covers "protocol or chain milestones" (§4.3.7). The spec does not explicitly assign it to DefiLlama or RpcMetrics.
   - What's unclear: Which oracle type handles "protocol/chain milestones" (e.g., "ETH staking ratio > 25%")?
   - Recommendation: Default to `RpcMetrics(3)` for on-chain milestones (active validator count, supply ratio) or `DefiLlama(2)` for TVL/volume milestones. Confirm with user before locking the adapterMap entry.
   - **RESOLVED:** `adapterMap[Event=2][ProtocolMilestone=8] = DefiLlama(2)` per Plan 02 (§4.3.7 protocol milestones are predominantly TVL/fees-based; DefiLlama already covers these). Owner can call `setAdapterMap(2, 8, RpcMetrics)` post-deploy without redeploying if on-chain validator/supply-ratio milestones are required.

2. **Criteria store insertion timing**
   - What we know: `criteriaText` arrives in `calls-preflight.ts` at call creation time. The `callId` is not known until the transaction is mined.
   - What's unclear: How to associate `criteriaText` → `callId` if the preflight stores it before the callId is known.
   - Recommendation: Use a `callCreatedAt` + `callerAddress` correlation key in the criteria store, or listen for `CallCreated` event and backfill. OR: for Phase 05.1 MVP, use the assetA-encoding approach for governance/NFT/CEX and defer the full criteria store to Phase 6.
   - **RESOLVED:** Governance + NFT calls use `assetA` encoding (no store lookup needed). DefiLlama / RpcMetrics / CexScraper calls use the `call_oracle_criteria` store (Option R1), populated via the `insertCriteria` wiring added in Plan 04 Task 2: a `CallCreated` event listener in `calls-preflight.ts` (or a settlement-watcher post-mine hook) calls `insertCriteria(callId, oracleType, identifier, targetUnit)` once the callId is known from the emitted event. Backfill for already-seeded Sepolia test calls is handled in the Plan 05 smoke task.

3. **NFT calls: adapterMap routing vs. assetA detection**
   - What we know: NFT calls use `MarketType.PriceTarget(0)` + `EventSubtype.None(0)` with `assetA` as an NFT contract address. `adapterMap[0][0] = Pyth(0)` is also used for regular price-target calls.
   - What's unclear: How does the watcher distinguish an NFT-TWAP call from a Pyth price call when both use the same (mT=0, eS=0)?
   - Recommendation: The watcher checks whether `assetA` encodes a Pyth feed key or an NFT contract address (the NFT check is already present in lines 486-515). This per-call heuristic is fine for Phase 05.1 MVP; the adapterMap could add a separate MarketType for NFT calls in a future phase.
   - **RESOLVED:** The settlement-watcher's existing `assetA` heuristic (lines 486-515) is authoritative for Phase 05.1 MVP. `adapterMap[0][0]` stays `Pyth(0)` so the Pyth price-target spine is completely unaffected. A dedicated NFT MarketType is a future-phase item and is listed as a Deferred Idea.
