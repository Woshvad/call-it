<objective>
Close Gap B.3: add the call_oracle_criteria Drizzle table + migration, implement resolveCriteria + insertCriteria helpers, wire insertCriteria into the call-creation flow (calls-preflight.ts), and integrate resolveCriteria into the non-governance adapters (DefiLlama, RpcMetrics, CEX). Governance calls (Snapshot/Tally) use the assetA-encoding approach instead of the criteria store.

Purpose: The on-chain Call struct does not carry string identifiers (proposalId, protocolSlug, tokenSymbol). Without this store, DefiLlama, RpcMetrics, and CEX adapters have no way to get the protocolSlug or tokenSymbol at settlement time — call_oracle_criteria stays empty, and SC-4 end-to-end settlement cannot be demonstrated. The fail-safe is absolute: if resolveCriteria returns null, the adapter returns { ambiguous: true } and the settlement DEFERs. Task 2 ensures rows ARE populated at call-creation time, making end-to-end settlement demonstrable on Sepolia.

**Design decisions (documented for user review):**
- Governance (Snapshot/Tally): use assetA encoding — proposalId is stored as uint256 in the call's assetA field at creation time. Snapshot proposalIds are 32-byte hex hashes (fit in bytes32/uint256). Tally proposalIds are on-chain numeric IDs (fit in uint256). The settlement-watcher already reads call.assetA and passes it to the adapter. This eliminates the criteria store lookup for governance calls.
- DefiLlama / RpcMetrics / ProtocolMilestone: criteria store (Option R1). protocolSlug is a human-readable string (e.g., "uniswap") that cannot fit in uint256.
- CEX: criteria store. tokenSymbol is a short string.
- NFT-TWAP: assetA is uint256(address(nftContract)) — no criteria store needed.
- targetValue: adapters receive call.targetValue from the 19-field Call struct (restored in Plan 03 Task 1). targetValue MUST be passed through fetchAndAttest and used in resolveValueOutcome(observedValue, call.targetValue). If targetValue genuinely cannot be derived for a specific path, that path must return { ambiguous: true } and DEFER — never default to 0n and settle against zero.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| criteria-store resolveCriteria → Fly Postgres | Relayer reads string identifiers from DB; if DB is unavailable or has wrong data, adapters must DEFER |
| assetA → proposalId decoding | Governance calls extract proposalId from call.assetA; wrong encoding at call creation = wrong proposalId at settlement |
| calls-preflight.ts insertCriteria → criteria store | Write path: preflight inserts criteria row after on-chain confirmation; failure is non-fatal to call creation but means settlement will DEFER |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.1-04-01 | Spoofing | Criteria store manipulation — attacker inserts wrong protocolSlug for a callId | mitigate | Only the relayer inserts into call_oracle_criteria (no direct user write path); on-chain criteriaHash (keccak256 of criteriaText) is stored on CallRegistry — relayer can cross-verify the stored identifier against the on-chain criteriaHash; mismatch should return ambiguous |
| T-05.1-04-02 | Denial of Service | criteria store unavailable at settlement time | mitigate | Fail-safe: resolveCriteria returns null → adapter returns { ambiguous: true } → settlement DEFERs to dispute window; no mis-settlement occurs; per RESEARCH.md Gap B.3 risk note |
| T-05.1-04-03 | Tampering | Wrong proposalId from assetA decoding (encoding mismatch at call creation) | mitigate | Snapshot proposalIds are bytes32 hashes that fit in uint256; Tally proposalIds are numeric uint256; the encoding contract must be documented in the call-creation flow so call creators store the correct value in assetA; validation is deferred to Phase 6 staging soak |
| T-05.1-04-04 | Tampering | Drizzle migration schema mismatch (types not matching schema.ts) | mitigate | db:generate is run from schema.ts, so the SQL type exactly matches the TypeScript definition; the BLOCKING checkpoint verifies the generated SQL + live table against expected columns |
| T-05.1-04-05 | Tampering | targetValue defaulted to 0n causing incorrect outcome resolution | mitigate | Adapters use call.targetValue from the 19-field ABI directly; 0n with no meaningful threshold triggers ambiguous return (not settlement against zero); per design decision note in Task 1 action |
</threat_model>
