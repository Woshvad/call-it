<objective>
Close Gap B.3: add the call_oracle_criteria Drizzle table + migration, implement resolveCriteria + insertCriteria helpers, and wire them into the non-governance adapters (DefiLlama, RpcMetrics, CEX). Governance calls (Snapshot/Tally) use the assetA-encoding approach instead of the criteria store.

Purpose: The on-chain Call struct does not carry string identifiers (proposalId, protocolSlug, tokenSymbol). Without this store, DefiLlama, RpcMetrics, and CEX adapters have no way to get the protocolSlug or tokenSymbol at settlement time — they would stub with an empty string and either settle incorrectly or fail. The fail-safe is absolute: if resolveCriteria returns null, the adapter returns { ambiguous: true } and the settlement DEFERs.

**Design decision (documented for user review):**
- Governance (Snapshot/Tally): use assetA encoding — proposalId is stored as uint256 in the call's assetA field at creation time. Snapshot proposalIds are 32-byte hex hashes (fit in bytes32/uint256). Tally proposalIds are on-chain numeric IDs (fit in uint256). The settlement-watcher already reads call.assetA and passes it to the adapter. This eliminates the criteria store lookup for governance calls.
- DefiLlama / RpcMetrics / ProtocolMilestone: criteria store (Option R1). protocolSlug is a human-readable string (e.g., "uniswap") that cannot fit in uint256.
- CEX: criteria store. tokenSymbol is a short string.
- NFT-TWAP: assetA is uint256(address(nftContract)) — no criteria store needed.

**BLOCKING migration task is included in this plan.**

Output: call_oracle_criteria table in Fly Postgres; criteria-store.ts module; criteria-store.test.ts passing; adapters wire in resolveCriteria calls.
</objective>
