<objective>
Vertical slice G: the user-facing feed (`/`) and profile shell (`/profile/[address]`) plus Profile Settings — completing the 4 baseline pages spec'd for Phase 1. The feed is driven by a relayer endpoint with an 800ms subgraph→polled-events race (D-24) and 10s Redis cache on the first page (D-26); the profile shell resolves the handle server-side via ENS reverse-record with a 24h Redis cache (D-13) AND falls back to Twitter handle → Farcaster handle → truncated 0x per AUTH-11. The frontend never sees the subgraph studio API key (D-27).

Purpose: After Plan 08, a user can publish a call but cannot yet see it back in a feed or visit their profile to see their history. This slice completes the Phase 1 critical-path receipt loop end-to-end at the UI level: publish → see your call in feed within 30s (subgraph indexing latency) → visit your profile to confirm. Plan 06's onboarding redirect requires `/` to exist as a destination. Plan 07's AddressBookManager has no home until Profile Settings ships.

Output: Working `/` feed (with empty-state, 800ms race, 10s cache, 5s polling), `/profile/[address]` (with ENS resolution + counts), `/profile/[address]/settings` (reusing AddressBookManager + Custody Disclosure + Export + Handle edit); relayer endpoints `/api/feed` + `/api/profile/:address`; Playwright e2e covering both shells.
</objective>
