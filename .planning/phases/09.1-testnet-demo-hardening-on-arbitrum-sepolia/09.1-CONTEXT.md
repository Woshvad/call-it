## Implementation Decisions

### Share-receipt loop & OG cards (fully discussed)
- **D-01:** ALL OG receipt card variants must be flawless when shared: CALLED IT, LOUD AND WRONG, CONTRARIAN HIT (requires wiring `fadeRealShare` from the subgraph — `og/[callId]/route.ts` currently hardcodes 0 making the variant unrenderable), CALLER EXITED (real time-decayed slash replaces the fabricated ~50% estimate, WR-10), the live card, and the fallback. Judges may click anything on the tape.
- **D-02:** Full prototype redesign of the OG card templates to the 09.2 design canon — Archivo display type, cream `#F5F1E8` / chartreuse `#E8F542` win / soft-red `#F87171` loss palette, hard black offset shadows. Satori flexbox-only (grid ban stands). 200px thumbnail baselines regenerated against the new design. This closes the D-04 (09.2) UI/OG divergence.
- **D-03:** Uniform handle policy (resolves CR-01/CR-02): unregistered wallets show a truncated address (`0xAB12…9F34`) on ALL surfaces — feed, leaderboard, OG cards — honest attribution beats suppression. OG routes STOP trusting forgeable `?handle=` (and any identity-bearing) query params: identity is resolved server-side from the callId via on-chain/subgraph data. `?as=fader` survives only as a viewer-lens flip (D-09 per-viewer render), never as an identity override.
- **D-04:** Acceptance gate = live end-to-end share proof: real receipt links shared on X AND Farcaster from the deployed app, cards rendering correctly in both timelines, plus Twitter Card Validator pass on every variant and regenerated 200px goldens. The demo path is the test. (Also fix WR-09 while in there: settled OG card scales targetValue 1e8 while the composer writes 1e6 — shared targets can render 100× off.)

### Browse surfaces with real data (fully discussed)
- **D-05:** BUILD two data rails only: (a) **feed card essentials** — pool $ per side, follow/fade counts, caller rep/accuracy on live feed cards (FollowFadeMarket reserves + subgraph aggregates mostly exist); (b) **price ticker** — a price endpoint (relayer already fetches Hermes spot prices for the trivially-true guard — reuse `hermes-spot.ts`) and flip the `NEXT_PUBLIC_SYSTEM_TICKER` flag ON (32px tape, D-11 of 09.2).
- **D-06:** NOT building in 09.1: profile/leaderboard depth (sparklines, 7d deltas, calibration/ROI, category bars, follower counts) and platform totals/hero StatBlocks. These stay hidden per the standing D-07 (09.2) honest-degradation rule.
- **D-07:** Feed tabs stay Live/Settled exactly as shipped ("I like it the way it is currently"). No Following tab, no Duels tab. Duels remain reachable via call pages and direct links.
- **D-08:** Sparse profile/leaderboard surfaces stay as-is ("I like it the way it is") — no layout tightening, no designed empty-state copy. Zero frontend work on sparse states.

### Standing invariants (carried forward, not re-discussed)
- **D-09:** 09.2's D-07 (honest degradation — hidden, never faked) and D-08 (no dead controls) remain law for every new surface and data rail.
- **D-10:** The 09.2 review triage ledger (WR-01/02/03/04/06/11/12 + IN-01..08 — viewer-handle plumbing, disputes admin gate, duel preflight, onboarding handle persistence, info cleanups) is dispositioned to this phase per 09.2-REVIEW.md; planner decides batching/ordering. ## Existing Code Insights

### Reusable Assets
- `hermes-spot.ts` (relayer): never-throw `getSpotPrice1e8` with exact expo normalization — the ticker endpoint is mostly a thin route over this.
- FollowFadeMarket reserve reads + subgraph entities already power the call page's pro-rated positions — feed-card pool $/counts can draw from the same sources.
- `useFeedMarketData.ts` shows the sanctioned read pattern: batched, chainId-pinned, staleTime-only (zero refetchInterval — test-pinned).
- OG pipeline (Phase 7) is Node-runtime @vercel/og with 200px golden tests — regeneration workflow already exists from 09.2 wave 9.

### Established Patterns
- Honest degradation (D-07) and no-dead-controls (D-08) are grep/test-pinned — new feed-card fields must render only when their data resolves.
- Source-assert test style (presentation-sweep, live-call-card pins) — new invariants (e.g., OG no-?handle-trust) should get the same pin treatment.
- Live-ops authorization split: pushes to master and Fly/Vercel deploys require explicit per-batch operator authorization; relayer deploys from a clean worktree.

### Integration Points
- Subgraph: any new aggregate (fade share, follow/fade counts) lands in the mappings → requires a Studio version bump + `SUBGRAPH_URL` env rolls (web Vercel + relayer Fly).
- Relayer: new public endpoints (ticker, feed enrichment) ride the existing Fastify route + CORS allowlist setup.
- Live share proof (D-04) needs the deployed origin — localhost is CORS-blocked by design.

</code_context>
