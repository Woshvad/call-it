# Phase 09.2: Code Review Report

**Reviewed:** 2026-06-10T18:55:07Z
**Depth:** standard
**Files Reviewed:** 79
**Status:** issues_found

## Summary

Reviewed the full Phase 09.2 neobrutalist UI rebuild surface (app pages, shell chrome, /new composer, OG routes, packages/ui primitives/compounds, token layer, and the lockstep Playwright/Vitest tests). The retheme itself is disciplined: flexbox-only holds in packages/ui and both OG render paths (the page-level `.layout`/`.activity-row` grids are explicitly allowed), outcome words consistently route through `lib/outcome-word.ts`, the locked win/loss hexes (#E8F542 / #F87171) are honored on the receipt surfaces, and degrade-to-hidden is mostly real.

However, four Critical findings stand: two AUTH-44 violations where truncated **wallet addresses are rendered as handles** (feed cards and both OG card routes — the OG route additionally accepts forgeable `?handle=` / `?as=fader` query params on official receipt imagery, the same class as the Phase 8 CR-01 fix), a **million-fold stake display error** in the `Receipt` preview (`formatStake` never divides bigint micro-USDC), and a **units mismatch** that feeds 18-decimal share balances into `FollowFadeModal`'s 6-decimal USDC `userPosition`, breaking position top-ups entirely. Several of these predate 09.2's markup-only mandate, but all live in files submitted for this review and contradict the phase's hard invariants.

Twelve warnings cover logic errors in the dispute-reversal preview gate, a wrong-handle comparison in the D-09 fader guard, duplicated criteria fields in the event composer, a tailwind token that still maps CONTRARIAN HIT's class to duel purple, fabricated/mis-scaled OG stats, and discarded user input (resolver note, onboarding handle).

## Critical Issues

### CR-01: Feed cards render a truncated wallet address as the caller handle (AUTH-44)

**File:** `apps/web/components/FeedList.tsx:40` (and `:52-55`)
**Issue:** When the relayer feed item carries no `displayHandle`/`handle`, the card handle falls back to `truncateAddress(item.caller ?? '')` — a shortened wallet address — which `CallCard` then renders as `@0x1234...abcd` on the public home feed. This directly violates the phase's hard invariant "handles only, never wallet addresses (AUTH-44)". The codebase elsewhere treats truncated-address handles as "no handle" (e.g., `AppShell.WalletPill` suppresses `source === 'truncated'`), so this fallback fabricates exactly what the rest of the app refuses to show.
**Fix:**
```ts
// FeedList.tsx — degrade to an anonymous label instead of an address fragment
const handle = item.displayHandle ?? item.handle ?? 'caller';
// or hide the handle row entirely when no handle exists (D-07 degrade-to-hidden)
```
Remove `truncateAddress` (it becomes dead code).

### CR-02: OG card routes render address fragments as handles AND accept forgeable identity/outcome query params

**File:** `apps/web/app/og/[callId]/route.ts:822` (also `:732-734`, `:866`); `apps/web/app/og/duel/[challengeId]/route.ts:551-552`
**Issue:** Two related defects on the official receipt imagery:
1. AUTH-44: when no `?handle=` hint is present, the card handle is `` `0x${callData.caller.slice(2, 8)}` `` (call route) / `` `0x${challenge.caller.slice(2, 8)}` `` (duel route) — an address fragment rendered as `@0x1a2b3c` on the public share card.
2. Forgeability: `?handle=` is attacker-controlled and rendered verbatim (`@${handle}`) onto a card that carries real on-chain stats for call N — anyone can mint `https://<app>/og/123?handle=vitalik` and circulate an official-domain image attributing the call to someone else. Likewise `?as=fader` flips the outcome word to `FADED CORRECTLY` for any viewer on any lost call with zero verification (`isViewerFader` is trusted straight from the URL at `:734`/`:866`). The product's core value is that receipts are *unfakeable*; Phase 8 fixed the identical class of bug (forgeable `?action=`, CR-01). The duel route resolves handles from ProfileRegistry on-chain — the call route should do the same instead of trusting the query string.
**Fix:** Resolve the caller handle server-side (ProfileRegistry `getProfile(callData.caller)` like the duel route, or the relayer profile endpoint), use `'someone'`/generic label when unresolvable (never an address slice), and either remove `?as=fader` or require a signed/HMAC parameter minted by the app when generating fader-specific share URLs.

### CR-03: Receipt preview displays stake 1,000,000x too large (missing 6-decimal division)

**File:** `packages/ui/src/compound/Receipt.tsx:75-78` (rendered at `:228-231`, `:297-299`)
**Issue:** `formatStake` does `typeof stake === 'bigint' ? Number(stake) : stake` with **no division by 1_000_000**. Both call sites (`apps/web/app/new/page.tsx:328-336`, `:461-469`) pass the RHF `stake` value, which is bigint micro-USDC (`MIN_STAKE` = 5_000_000n). The live `/new` preview and quote preview therefore show **"$5,000,000"** for a $5 stake, and the IF WRONG row shows "−$5,000,000". Compare `CallCard.formatStake` (`packages/ui/src/compound/CallCard.tsx:67-71`), which correctly divides bigint by 1e6 — the two primitives disagree on the same prop contract.
**Fix:**
```ts
function formatStake(stake: number | bigint): string {
  const n = typeof stake === 'bigint' ? Number(stake) / 1_000_000 : stake;
  return `$${n.toLocaleString('en-US', { minimumFractionDigits: 0, maximumFractionDigits: 2 })}`;
}
```

### CR-04: FollowFadeModal receives 18-decimal shares as its 6-decimal USDC `userPosition` — breaks top-ups and headroom math

**File:** `apps/web/app/call/[id]/page.tsx:1492-1495` (consumed at `packages/ui/src/compound/FollowFadeModal.tsx:128, 144, 315, 339, 364`)
**Issue:** The call page passes `userFollowPosition = followShares` / `userFadePosition = fadeShares` (raw 18-decimal share balances, acknowledged in the comment "use shares as proxy") into `FollowFadeModal`'s `userPosition` prop, which is documented and used as 6-decimal USDC: `remainingHeadroom = MAX_POSITION - userPosition` and the "Your Position ${...} / $100" display. For any user holding shares, `userPosition` is astronomically larger than `MAX_POSITION`, so `remainingHeadroom` goes negative → `amountTooHigh` is always true → **the Confirm button is permanently disabled for every existing position holder**, with a garbage "Max remaining headroom: $-..." message. The same file already knows the correct conversion (WR-01 at `:1233-1242` computes `shares * reserve / totalShares`).
**Fix:** Pass the converted USDC value:
```ts
const userFollowPosition = followTotalShares > 0n ? (followShares * followReserve) / followTotalShares : 0n;
const userFadePosition   = fadeTotalShares  > 0n ? (fadeShares  * fadeReserve)  / fadeTotalShares  : 0n;
```

## Warnings

### WR-01: D-09 fader-guard fallback compares against the CALLER's handle, not the viewer's

**File:** `apps/web/app/call/[id]/page.tsx:1184-1193`
**Issue:** The post-settlement fallback for `viewerIsWinningFader` is `finalPositions.some(p => p.side === 'fade' && p.handle === (callData?.handle ?? ''))` — `callData.handle` is the **caller's** handle, not the viewer's. Two consequences: (a) legitimate winning faders never see FADED CORRECTLY after FFM positions are cleared (the comparison can't match them), and (b) if the caller's handle ever appeared on the fade side, *every* authenticated viewer would see FADED CORRECTLY. The guard fails closed for the public (D-09 holds) but the feature is broken for its intended audience.
**Fix:** Compare against the viewer's resolved handle (e.g., from `useProfile(userAddress)`), or have the relayer's `/positions` payload include addresses for internal matching (never rendered).

### WR-02: Dispute reversal preview + two-step confirm can be bypassed entirely

**File:** `apps/web/app/disputes/page.tsx:223-242` (render gate `:409-440`, `:495-539`)
**Issue:** `loadReversalPreview` derives the current outcome from `state?.preview?.currentOutcome` — but `preview` is `null` before any fetch, so `currentOutcomeNum` defaults to `OUTCOME_CALLER_LOST`. When the owner's first interaction selects `CallerLost`, `isReversal` computes false → no preview is fetched → the render-side `isReversal` (`state.preview !== null` guard) is also false → the **single-click "Resolve · CallerLost" button is enabled with no reversal preview and no two-step confirm**, even when the on-chain outcome is CallerWon. This defeats the page's own stated contract ("REQUIRED before confirm — D-07").
**Fix:** Fetch the provenance snapshot on admin-section mount (so `currentOutcome` is known before any selection), and gate the confirm on `preview !== null` whenever the selected outcome differs from the *fetched* current outcome.

### WR-03: Resolver note is collected but never persisted

**File:** `apps/web/app/disputes/page.tsx:474-486` (submit at `:244-259`)
**Issue:** The "Resolver Note (public)" textarea writes `state.note`, but `handleResolve` only calls `resolveDispute(callId, outcome)` — the note is sent nowhere (no contract arg, no relayer POST). The resolved list renders `dispute.resolverNote` from the relayer, so the owner's typed rationale is silently discarded. Silent loss of user input on a destructive admin action.
**Fix:** POST the note to the relayer (e.g., `POST /api/disputes/:id/note`) after tx confirmation, or remove the input until a sink exists (D-08: no dead controls).

### WR-04: Duel accept preflight breaks when `callerStake` is 0 pre-accept

**File:** `apps/web/app/duel/[challengeId]/page.tsx:398-427` (vs. comment at `:613-616`)
**Issue:** The CR-04 fix comment states "In a pre-accept (Proposed) state callerStake is 0n". If so, `callerMatchingStake = min(0, challengerStake) = 0n`, which makes `callerNeedsApproval = allowance < 0n` always false and `handleCallerApprove` early-return (`if (!callerMatchingStake) return`). The caller is shown ACCEPT directly, and `acceptChallenge` will revert on-chain when the escrow pulls the matching stake without allowance (T-3-06-06 preflight defeated). One of the two assumptions (pot math vs. preflight math) is wrong about the same field.
**Fix:** Base the matching stake on `challengerStake` when `callerStake === 0n` in Proposed state: `const callerMatchingStake = liveState ? (liveState.callerStake > 0n ? min(callerStake, challengerStake) : liveState.challengerStake) : 0n;`

### WR-05: Duplicate "Resolution Criteria" field rendered twice for event-type calls

**File:** `apps/web/app/new/page.tsx:404-406` (with `apps/web/app/new/components/EventFields.tsx:143-147`)
**Issue:** For `marketType === 'event'`, `EventFields` renders its own `<CriteriaField isRequired={requiresCriteria}>` AND the page renders a second `<CriteriaField isRequired={false}>` (`marketType !== 'priceTarget'` is true for both `spreadVs` and `event`). Two textareas bound to the same `criteriaText` RHF field appear on the form, with contradictory required indicators. In quote mode the "Your thesis" textarea makes it a third writer to the same field.
**Fix:** Render the page-level `CriteriaField` only for `marketType === 'spreadVs'`:
```tsx
{marketType === 'spreadVs' && <CriteriaField control={control} errors={errors} isRequired={false} />}
```

### WR-06: Event subtype select displays an option that doesn't match form state

**File:** `apps/web/app/new/components/EventFields.tsx:74-85`
**Issue:** Default `eventSubtype` is `'none'` (page defaultValues), but the select filters `'none'` out of its options. The browser displays the first option ("TVL Milestone") while the form value remains `'none'` until the user actively changes it — so a user can submit believing TVL Milestone is selected while preflight receives `none`, and the criteria-required logic (`CRITERIA_REQUIRED_EVENT_SUBTYPES.has('none')` → false) is computed for the wrong subtype.
**Fix:** Either set `eventSubtype` to a valid default when switching to event mode (`setValue('eventSubtype', 'tvlMilestone')` in `MarketTypeSwitcher`/an effect) or include a disabled "Select event type" placeholder option bound to `'none'`.

### WR-07: On-chain metric value only replaces the FIRST space

**File:** `apps/web/app/new/components/EventFields.tsx:115`
**Issue:** `metric.toLowerCase().replace(' ', '_')` replaces a single space — "Daily Active Users" becomes `"daily_active users"` (embedded space remains). The malformed value is written to `assetB` and flows into preflight/dup-check/on-chain metadata.
**Fix:** `metric.toLowerCase().replaceAll(' ', '_')` (or `.replace(/ /g, '_')`).

### WR-08: `text-outcome-contrarian` Tailwind token still resolves to duel purple, contradicting T-04-07-03

**File:** `packages/ui/tailwind.preset.ts:43` (vs. `packages/ui/src/tokens/colors.ts:23` and `Stamp.tsx:49-55`)
**Issue:** The preset maps `'outcome-contrarian': 'var(--accent-duel)'` (#A855F7) while `COLOR_MAP['outcome-contrarian']` is #E8F542 (D-03 fix). `Stamp` applies BOTH the class `text-outcome-contrarian` and an inline `color` style; the inline style wins today, so the stamp renders correctly — but any consumer using the class alone (current or future) renders CONTRARIAN HIT in purple, violating the locked-color invariant (T-04-07-03: "CONTRARIAN HIT = #E8F542, NOT purple"). Two sources of truth disagree under the same token name.
**Fix:** Map the preset key to the win color (`'outcome-contrarian': 'var(--accent-win)'`); if duel purple needs a utility, it already has one (`accent-duel`).

### WR-09: Settled OG card target value scaled 1e8 while the composer writes 1e6

**File:** `apps/web/app/og/[callId]/route.ts:869-872` (with `apps/web/app/new/components/PriceTargetFields.tsx:56-62`)
**Issue:** The OG route formats the on-chain `targetValue` via `formatOraclePrice` (divides by 1e8, "1e8 oracle scale"), but the /new composer stores price targets at 1e6 scale (`BigInt(Math.round(usd * 1_000_000))`). If both touch the same on-chain field, one of the two is off by 100x — an $80,000 target written by the composer renders as "$800.00" on the settled share card (or the composer is writing wrongly-scaled targets on-chain). Cross-module scale contract needs a single authority.
**Fix:** Pin the target scale in `@call-it/shared` (one constant, one formatter) and use it in both the composer and the OG route; verify against the relayer preflight's expected scale.

### WR-10: CallerExited OG card fabricates the "STAKE SLASHED" stat (~50% estimate)

**File:** `apps/web/app/og/[callId]/route.ts:910`
**Issue:** `stakeSlashed: formatUsdc(BigInt(stakeRaw / 2n))` presents a hardcoded 50% estimate as a definite dollar figure on a permanent share card. The actual penalty is time-decayed (15%→50%). This violates the phase invariant "degrade-to-hidden for missing data (no fabricated numbers)" — the exact slash exists in the `CallerExited` event/subgraph and is even fetched for the in-app banner (`callerExitedPenalty`).
**Fix:** Read the real penalty from the subgraph (like `settledFields.repDelta` at `:903-904`), and omit the stat cell entirely when unavailable.

### WR-11: Quote success Share button tweets the literal placeholder "@you"

**File:** `apps/web/app/new/components/QuoteSuccess.tsx:124-129`
**Issue:** `ShareButton` is invoked with `handle="@you"`, so the composed tweet text (via `buildShareText`) contains the placeholder string "@you" instead of the user's real handle — a real share intent, not a preview. Also `outcomeWord="CALLED A QUOTE"` is not a `lib/outcome-word.ts` word, and `receiptUrl` points at the *parent* call rather than the new quote call.
**Fix:** Pass the viewer's resolved handle (from `useProfile`) or omit the handle from the share text when unknown; link the receipt URL to the new quote-call id once available from the publish result.

### WR-12: Onboarding handle screen discards the typed handle

**File:** `apps/web/app/onboarding/handle/page.tsx:79-94`

**Fix:** Persist the handle on submit (relayer onboarding endpoint or `ProfileRegistry.setDisplayHandle` via the existing wagmi path), or relabel the field as a preview of the auto-derived handle if persistence is intentionally deferred.

### IN-08: Misleading dead state in quote-success wiring
