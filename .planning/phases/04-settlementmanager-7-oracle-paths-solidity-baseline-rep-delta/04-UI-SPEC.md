## ⚠ Design-System Conflict Found (READ FIRST — load-bearing for the outcome hero)

Detection surfaced a **real, shipped divergence between two color palettes** in the codebase. This matters because the outcome-word hero IS the Phase 4 deliverable and one palette violates a LOCKED requirement.

| Layer | Palette | Win | Loss | Contrarian | Muted | Body text |
|-------|---------|-----|------|-----------|-------|-----------|
| `packages/ui` Tailwind preset + `tokens/colors.ts` (used by `<Stamp>`, `<Receipt>`, `<ConvictionBar>`) | coarse 9-color | `#22C55E` | `#EF4444` | **`#A855F7` purple** | `#A1A1AA` | `#FFFFFF` |
| App pages (Live Receipt page) + spec §14.1 + REQUIREMENTS (UI-17/38, SHARE-06) | detailed §14.1 | `#4ADE80` | `#F87171` | **`#E8F542` accent** | `#94A3B8` | `#F1F5F9` / `#E8E8E8` |

**The LOCKED requirements win.** UI-17, SHARE-06, §15.7, and §16.3 unambiguously state: `CONTRARIAN HIT = accent #E8F542` (NOT purple), `CALLED IT = #4ADE80`, `LOUD AND WRONG = #F87171`, `COLD CALL = #94A3B8`. The existing `<Stamp>` primitive's `outcome-contrarian` token (`#A855F7` purple) **directly violates UI-17** and must NOT be used for the Phase 4 outcome hero.

**Resolution (binding for this phase):**
1. The Settled Receipt outcome hero renders with the **detailed §14.1 palette via explicit hex** (the table in [Color](#color) below), matching the already-shipped Live Receipt page — NOT the `<Stamp>` `StampColor` token map.
2. The `<Stamp>` primitive is REUSED for its animation (overshoot scale) but the executor passes the outcome color as an explicit hex override; if `<Stamp>`'s `StampColor` union cannot express `CONTRARIAN HIT = #E8F542` / `COLD CALL = #94A3B8` / `FADED CORRECTLY = #E8F542` cleanly, extend the `Stamp` `color` prop to accept a raw hex (small, additive change — see [Primitive Reuse + Extension Map](#primitive-reuse--extension-map)). Do NOT change the existing `outcome-*` tokens (other phases depend on them).
3. This conflict is recorded so the checker does not flag the Phase-4 hex usage as "off-palette" — the hex IS the locked palette; the preset token is the stale one. A follow-up to reconcile `tokens/colors.ts` to §14.1 is noted as an upstream dependency (out of Phase-4 scope — touching it risks Phase 1/2/3 primitives).

### Weight System (2 weights only)

| Weight | Value | All uses |
|--------|-------|----------|
| Regular | 400 | Body copy, data values, meta/label copy, timestamps, dispute thread running text, provenance raw-data values |
| Bold | 700 (Display/Hero use 800 "black") | Outcome hero word, call statement, section labels (uppercase tracked 0.12em), CTA button labels, handle names, lozenge text, dispute-block headings, P&L emphasis |

### Typography Rules

- **Uppercase labels:** `letter-spacing: 0.12em` per §14.6 / UI-46 — applies to all 12px uppercase Space Grotesk labels ("FINAL POSITIONS", "FINAL VALUE", "TARGET", "CONVICTION", "P&L", "SETTLED FROM", "PENDING DISPUTE", "CONTRARIAN", "FADER WIN", "OPEN DISPUTES", "RESOLVED", "ORACLE PROOF").
- **JetBrains Mono at 12px** for all numeric/hash data: tx hashes, oracle price/confidence, rep deltas, P&L, bond `$5.00`, signature bytes (truncated to first 10 + last 8 chars with middle ellipsis).
- **Outcome hero is Syne 96px** (UI-16) — this is the ONE place the 4-size scale is exceeded, and it is LOCKED by the requirement. The 200px-readability gate (UI-18/SHARE-12) is measured against this rendering AND its OG-card equivalent.
- **OG card type is inline style** (Satori cannot use Tailwind class names). Mirror sizes/weights as numeric style props, matching the existing `/og/[callId]` and `/og/duel/[challengeId]` routes.
- **`display: flex` only — never `display: grid`** — applies to ALL Phase 4 layouts including OG cards and Tailwind page layouts (Pitfall 15 + ESLint `no-display-grid` + CI grep guard).

They do NOT belong in this visual contract. This contract specifies only how each word RENDERS. The planner must define the thresholds (lean toward more wins earning the celebratory `CONTRARIAN HIT` per the founder's steer) and the page/OG layers consume the resulting word + color.

**Accent (`#E8F542`) reserved for (explicit list — never "all interactive elements"):**
1. The 4px corner brackets on the page frame + FINAL POSITIONS block + provenance modal (UI-14, UI-21)
2. `CONTRARIAN HIT` and `FADED CORRECTLY` outcome words (+ their 3–4px border and hard offset shadow)
3. The target value highlighted inside the Settled-receipt call statement (UI-15)
4. "[ SHARE THE RECEIPT → ]" primary CTA fill (black text on accent — UI-20)
5. "view oracle proof ↗" provenance link + the Arbiscan tx-hash link in the proof modal (SETTLE-52 / D-10)
6. Positive rep-delta count-up digits share `#4ADE80`, NOT accent — accent is reserved; do not tint rep deltas accent
7. OG cards (variants 2 & 4): wordmark "CALL IT", corner brackets, accent-outcome words, footer brand

**Destructive / caution colors:**
- `#F87171` (loss red): `LOUD AND WRONG` hero; negative rep/P&L; the "Yes, reject"-pattern destructive confirm on owner dispute-resolution when it would forfeit a disputer bond; "bond forfeited to treasury" warning copy.
- `#FB923C` (amber): the Caller-Exited banner and the "PENDING DISPUTE" block. Amber is caution/in-progress, NOT destructive — it signals "this receipt is not in its final settled state."

## Design Contract Notes

1. **Outcome palette = §14.1 hex, NOT the `<Stamp>` purple token.** See the [conflict section](#-design-system-conflict-found-read-first--load-bearing-for-the-outcome-hero). `CONTRARIAN HIT = #E8F542` (accent), NOT `#A855F7` (purple). The checker must treat the §14.1 hex as authoritative for Phase-4 outcome rendering; the preset's `outcome-contrarian` purple is the stale token and is out of scope to fix this phase.

2. **Flexbox only in ALL Phase 4 surfaces** — Pitfall 15 + ESLint `no-display-grid` + CI grep. Verify `grep -rn "display.*grid\|gridTemplate" apps/web/app/call/ apps/web/app/og/ apps/web/app/disputes/` returns 0. Two-column blocks (FINAL POSITIONS, dispute thread, duel OG) are `flex-direction: row`.

3. **Wallet address NEVER appears** (AUTH-44) — the Settled receipt, all OG variants, the dispute modal, the provenance modal, and the `/disputes/` log show handles only. The `<Receipt>` type already excludes `address`; the shipped page comments the caller address as internal-only. This propagates to every Phase 4 surface.

4. **Per-viewer is page logic, not a layout variant (D-09)** — the outcome WORD is computed for the current viewer at render time. The default SHARED OG card is caller-centric (4 caller words only); a winning fader generates their own `?as=fader` card. There is one shared image, kept unambiguous.

5. **The trust surfaces are the product — build them FULLY (D-06/D-07/D-10).** The full self-serve dispute system (in-app raise + IPFS evidence + public log + owner resolve with reversal preview) AND the full provenance proof modal (raw oracle data + EIP-712 signature) are the "unfakeable, verifiable receipt" core value made tangible, and the strongest demo beats. Do NOT trim them to manage phase size.

6. **D-08 thresholds are NOT in this contract** — the % cutoffs for `CONTRARIAN HIT` vs `CALLED IT` and the `COLD CALL` rep-delta definition are planner/rep-math discretion (derive from consensus + rep-delta signals; lean toward more wins earning `CONTRARIAN HIT`). This contract specifies rendering only. Flagged as an upstream dependency for the planner.

7. **The 96px outcome hero is the single type-scale exception** — LOCKED by UI-16. The 200px readability gate (UI-18/SHARE-12) is measured against the OG-card render (which may drop to ≥64px to keep the longest word on one line) — the page hero stays 96px (more horizontal room). Phase 4 renders + visually QAs the cards; the formal 200px QA *gate* + Twitter Card Validator are a Phase 7 finalization (per CONTEXT out-of-scope).

8. **Desktop-first; mobile deferred to Phase 9.** The Settled Receipt IS a Phase-9 critical mobile page (the share-link landing target), but the responsive pass is Phase 9. Phase 4 writes desktop CSS (≥1024px content). Action-row buttons specify the stacked-mobile contract for Phase 9 to consume.

9. **OG routes branch, they do not fork.** Variants 2 (settled) and 4 (caller-exited) live in the SAME `/og/[callId]/route.ts`, branching on `callData.status` — not new routes. The duel variant-3 fill is a flag-flip in the existing `/og/duel/[challengeId]/route.ts`. This keeps the cache-bust (`?v={statusVersion}`) and `renderFallback` plumbing unchanged.

10. **Type scale rationale** — 12 / 16 / 28–32 / 48 px maps to §14.6: 12px = all meta/data/label with 0.12em tracking; 16px = body + interactive; 28–32px = section/modal headings; 48px = call statement + amber heroes. The 96px outcome word is the locked apex (UI-16). No standalone 64px on the page (folded into 48px display + 96px hero), consistent with the Phase 3 type-scale decision; OG cards may use intermediate fit-to-line sizes (64–96px) since Satori cannot wrap heroes gracefully.
