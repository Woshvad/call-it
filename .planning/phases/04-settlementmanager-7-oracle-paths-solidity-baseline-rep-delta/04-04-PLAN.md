<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Relayer → SettlementManager.settle() | walletClient (GCP KMS key); settle() is permissionless but relayer is the reliable actor |
| Relayer → SettlementManager.submitAttestation() | EIP-712 signed by per-type KMS key; chainId=42161n prevents cross-chain replay |
| Hermes API → relayer | Public pull oracle; VAA cryptographically verified on-chain by Pyth contract |
| DefiLlama API → relayer | Unauthenticated REST; relayer signs attesting data with per-type KMS key; on-chain ecrecover verifies |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Spoofing | Fake relayer attestation (compromised defillama key) | mitigate | Per-type KMS key separation (Pitfall 7); defillama key can only forge DefiLlama attestations, not NFT-TWAP or CEX — limits blast radius |
| T-04-04-02 | Repudiation | Cross-chain EIP-712 replay | mitigate | EIP-712 domain chainId=42161n + verifyingContract=SETTLEMENT_MANAGER_ADDRESS; attestation from one chain rejected on another (Pitfall 7) |
| T-04-04-03 | DoS | SettlementManager ETH balance exhausted | mitigate | pyth-adapter.ts checks balance before settle; alerts Telegram at < 0.01 ETH; OPS-15 runbook covers top-up (Pitfall 4) |
| T-04-04-04 | DoS | Pyth Hermes unavailable | mitigate | 30×60s retry loop (SETTLE-10); after exhaustion → dispute window opens (SETTLE-11); Telegram alert fires |
| T-04-04-05 | Information Disclosure | KMS private key exposure | mitigate | gcpKmsAccount — private key never leaves GCP KMS; all signing is remote (D-05) |
</threat_model>
