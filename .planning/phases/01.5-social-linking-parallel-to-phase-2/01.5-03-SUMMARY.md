# Phase 01.5 Plan 03: VerifiedBadge + AUTH-09 surfaces + AUTH-10 invariant Summary

Reusable `<VerifiedBadge>` primitive (VERIFIED · X / · FC / · X · FC, null when unlinked) reusing the neobrutalist Tag, wired onto ProfileHeader / CallCard / Receipt with a typed badge-host seam for Duel/Leaderboard, plus an explicit AUTH-10 zero-mechanical-effect parity test proving verified and unverified users get byte-identical fee/stake/rep treatment.

## Accomplishments

- **VerifiedBadge primitive (AUTH-09):** Derives the combined label from `verifiedX`/`verifiedFc`; renders `null` when both are false/undefined; reuses `Tag intent="warning"` (brand-accent #E8F542) with `data-testid="verified-badge"`; inline-flex (never grid) preserved for Satori/OG (Pitfall 15). Exported (`VerifiedBadge` + `VerifiedBadgeProps` + `VerifiedBadgeHost`) from the `@call-it/ui` barrel.
- **AUTH-09 surface wiring:** ProfileHeader (replaced the inline VERIFIED span; added `verifiedX`/`verifiedFc`, kept deprecated `verified?` mapping to `verifiedX`), CallCard (added fields to `CallCardData`, badge next to `@handle`), Receipt (added fields to `ReceiptData`, header badge host next to the caller handle — directly feeds the parallel Phase 2 Live Receipt; Settled Receipt wiring stays Phase 4 per D-07).
- **Duel/Leaderboard badge-host seam (D-07):** No `DuelParty`/`LeaderboardRow` presentational component exists in `packages/ui` yet, so per the plan the seam is the exported `VerifiedBadgeHost` type carrying `verifiedX?`/`verifiedFc?` with a "full page Phase 3/7" comment. The full pages were NOT built.
- **AUTH-10 zero-mechanical-effect invariant (D-09):** A parity test that (1) runs the actual fee BPS constants, stake/position bounds, conviction floor, and caller-exit rep-delta/penalty/slash math from `@call-it/shared` through `mechanicalInputsFor()` and asserts verified vs unverified (and X-only/FC-only/both) snapshots are deep-equal, and (2) statically guards that `fees.ts`, `call-gates.ts`, and `follow-fade-gates.ts` contain no `verifiedX`/`verifiedFc`/social-verification reference. A non-trivial guard asserts concrete non-zero fee values so the parity is not vacuous.

## Decisions Made

1. **Reuse Tag, no new styling** (D-07 discretion) — `intent="warning"` is the brand-accent treatment; keeps the badge inline-flex for Satori.
2. **`verified?` kept as deprecated back-compat on ProfileHeader** — existing callers passing `verified` continue to work (maps to `verifiedX`), avoiding a breaking change; new callers use the richer flags.
3. **VerifiedBadgeHost is a type seam, not a component** — no Duel/Leaderboard presentational component exists in `packages/ui`; the plan's fallback ("typed field on the relevant exported data type") was applied as a dedicated exported type. Full pages remain Phase 3/7.
4. **AUTH-10 test passes immediately and that is correct** — it is a regression-guard invariant for a property the codebase already holds (verification never touched mechanics). Per the TDD fail-fast note this was investigated: verification flags live only in presentational/profile-enrichment paths, so the guard correctly locks the existing-correct state rather than driving new code.
