<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → relayer link route | Session-gated; the handle/fid must come from verified proof, never the request body |
| relayer → ProfileRegistry | KMS oauth-proof wallet is the only authorized linker (msg.sender == relayer) |
| relayer → external (Privy / Farcaster relay) | Proof verification crosses to Privy + Farcaster relay before any on-chain write |
| follow_graph / Redis cache → other users | Viewer-only; never expose another user's graph (AUTH-17) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01.5-02-01 | Spoofing | Twitter handle source | mitigate | Derive handle from Privy-verified linkedAccounts, never from request body (Pitfall 2; twitter-proof test asserts) |
| T-01.5-02-02 | Spoofing/Tampering | SIWF message | mitigate | Server-issued nonce (Redis TTL) + strict domain equality in verifySignInMessage (Pitfall 3) |
| T-01.5-02-03 | Spoofing | Handle squatting | mitigate | social_link_index partial-unique active index; second account → 409 (D-06) |
| T-01.5-02-04 | Information Disclosure | follow_graph leakage | mitigate | Viewer-only, keyed by privyUserId; routes never accept a target-user param (AUTH-17) |
| T-01.5-02-05 | Information Disclosure | Stale data after unlink | mitigate | Dual-store purge + SocialUnlinked watcher backstop (D-13, Pitfall 6; unlink-purge test asserts both stores) |
| T-01.5-02-06 | Information Disclosure | PII on-chain | mitigate | proofHash is keccak of claims; no raw PII written (D-05) |
| T-01.5-02-07 | Elevation of Privilege | Relayer key | mitigate | GCP-KMS custody (no local key bytes); setRelayer owner-rotatable (01.5-01) |
