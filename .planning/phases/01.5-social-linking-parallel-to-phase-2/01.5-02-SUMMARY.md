# Phase 01.5 Plan 02: Relayer Social-Link Service Summary

**Relayer social-link backbone: Privy-verified Twitter + SIWF Farcaster proofs submitted to ProfileRegistry.linkTwitter/linkFarcaster from the KMS oauth-proof wallet, off-chain handle uniqueness (D-06), structured PII-free proofHash (D-05), dual-store unlink purge with SocialUnlinked backstop (D-13/AUTH-17), and /api/profile verifiedX/verifiedFc derived from subgraph handles (D-08).**

## Accomplishments

- **Task 1 — proofHash lib + proof verifiers + schema (`451df7d`):**
  - `social-proof-hash.ts`: `computeProofHash(user, platform, handle, subject, issuedAt, nonce)` via viem `keccak256(encodeAbiParameters(...))` (D-05) + `generateNonce()` (32-byte hex). Deterministic, field-sensitive, 0x+64-hex output, no readable PII.
  - `twitter-proof.ts`: `extractVerifiedTwitter(privyUserId)` reads the Privy-verified `twitter_oauth.username` + `subject` server-side; throws `twitter_not_linked_in_privy` when absent; arity-1 (handle can never come from the body — Pitfall 2).
  - `farcaster-proof.ts`: `verifyFarcasterProof({ nonce, domain, message, signature })` wraps `@farcaster/auth-client` `verifySignInMessage`; rejects empty nonce/domain (`siwf_missing_nonce_or_domain`) before calling the verifier (Pitfall 3); throws `siwf_verification_failed` on `success:false`; returns `{ fid }`.
  - `schema.ts`: appended `follow_graph` + `social_link_index` (unique index `social_link_active_idx` on `(platform, handle_normalized)`); drizzle migration `0005_warm_ego.sql` generated (NOT applied — operator/live step).
  - Installed `@farcaster/auth-client@0.7.1` (the one CORE dep). 19 tests pass.

- **Task 2 — social-link routes + KMS submit (`b83f775`):**
  - `social-link-index.ts`: `assertHandleAvailable` (409 `HandleAlreadyLinkedError` if active elsewhere — D-06), `recordActiveLink` (reactivate-on-relink), `markUnlinked`, `normalizeHandle` (lowercase + strip leading `@`).
  - `oauth-proof-submitter.ts`: KMS-backed `writeContract` mirroring the settlement-watcher walletClient interface — serializes the EIP-1559 tx, signs the digest via the new `account.signDigest`, broadcasts via `sendRawTransaction` (D-03).
  - `kms-signer.ts`: exposes `account.signDigest` (`GcpKmsAccount` type) so raw tx signing works despite `signTransaction` throwing by design.
  - `social-link.ts`: 3 session-gated POST routes — `/api/social/link` (Twitter), `/api/social/link/farcaster` (SIWF nonce checked against a Redis-issued nonce), `/api/social/unlink-purge` (dual-store clear). zod V5 bodies, 50-byte handle cap, structured pino V7 logs, Pitfall 5 (failures never throw into the session). Registered in `index.ts`. 9 tests pass.

- **Task 3 — backstop watcher + profile enrichment (`a5bb3ad`):**
  - `social-unlink-watcher.ts`: polls `SocialUnlinked(address indexed user, uint8 kind)` with chunked ≤9-block getLogs (mirrors notification-fanout WR-05), purges `follow_graph` + Redis + marks `social_link_index` unlinked; idempotent; zero-address idle guard; per-event failure isolation (D-13, AUTH-17). Started in `index.ts onReady`.
  - `subgraph-client.ts`: `queryProfileSocials(address)` → `{ twitterHandle, farcasterHandle }` from the Profile entity (D-08).
  - `profile.ts`: `verifiedX = !!twitterHandle`, `verifiedFc = !!farcasterHandle` derived from the subgraph (no longer hardcoded false); the read joins the existing `Promise.allSettled` and degrades to null/false on failure (Pitfall 5); a null handle (unlinked) correctly flips the flag to false.
  - Full relayer suite green (171 passed / 1 skipped); build exit 0.

## Decisions Made

1. **A2 confirmed (Privy getUser shape).** `@privy-io/server-auth@1.32.5` `getUser(userId)` returns `User` with `linkedAccounts`; the `twitter_oauth` account exposes `{ username: string|null, subject: string }`. Matches the plan interface block exactly — no identity-token-parse fallback needed.
2. **Farcaster on-chain handle = `fid:{fid}`.** No Neynar key in the CORE wave, so the verified fid is the canonical handle/subject; the FEED wave (later 01.5 plan) can resolve fid→fname. Keeps linkFarcaster fully functional now.
3. **Plain unique index + reactivate-on-relink** rather than a Postgres partial unique index. `recordActiveLink` updates the existing `(platform, handle_normalized)` row (clearing `unlinkedAt`, repointing `userAddress`) instead of inserting a duplicate, satisfying D-06 "one active link per handle" while keeping the schema portable.
4. **`account.signDigest` added to `gcpKmsAccount`** (a superset of the viem Account) + a bespoke `oauth-proof-submitter` `writeContract`, because `gcpKmsAccount.signTransaction` throws by design (the generic viem walletClient path can't KMS-sign raw txs).
5. **Injectable `{ db, getSubmitter }` on the route plugin** for unit mocking; `unlink-purge` never builds the KMS submitter (AUTH-12 unlink is user-callable on-chain — the relayer only purges off-chain data).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Added `account.signDigest` to `gcpKmsAccount` + `oauth-proof-submitter.ts`**
- **Found during:** Task 2 (KMS submission path).
- **Issue:** The plan's `<read_first>` flagged that `gcpKmsAccount.signTransaction` throws by design and said to "build a viem walletClient with this account or use serializeTransaction + the account signing path". The account's `signDigest` was a private closure inside `gcpKmsAccount`, not exported — so neither path was buildable without a code change.
- **Fix:** Exposed `signDigest` on the returned account (`Object.assign`, additive — existing consumers unaffected) and added `oauth-proof-submitter.ts` implementing the serialize→signDigest→sendRaw flow behind a `writeContract` method that mirrors the settlement-watcher interface.
- **Files modified:** `apps/relayer/src/lib/kms-signer.ts`, `apps/relayer/src/lib/oauth-proof-submitter.ts` (created).
- **Verification:** relayer build exit 0; the existing `kms-roundtrip` test still passes (skipped without GCP creds, but typechecks).
- **Committed in:** `b83f775` (Task 2).

**Total deviations:** 1 auto-fixed (blocking — KMS submit prerequisite). No architectural changes; no Rule 4 checkpoints. The on-chain `setRelayer` wiring + live tx remain operator/runtime concerns (deferred per objective) — all on-chain submission is unit-mocked here.
