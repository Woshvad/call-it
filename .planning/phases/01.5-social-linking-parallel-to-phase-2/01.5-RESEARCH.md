## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01: No new ProfileRegistry contract code.** Deployed ProfileRegistry already has `linkTwitter`/`linkFarcaster` (relayer-only), `unlinkTwitter`/`unlinkFarcaster` (user self-service), `SocialIdentity` storage (twitter/farcaster handle + proofHash + linkedAt; 50-byte handle cap), `relayer` + `setRelayer`, and `SocialLinked`/`SocialUnlinked` events. Phase 1.5 wires the relayer (`setRelayer`) and builds the service/frontend/data layers on top.
- **D-02: Shared ProfileRegistry redeploy with Phase 2.** Phase 2 (02-CONTEXT D-04) redeploys ProfileRegistry to add the `authorizedRepWriters` set. That redeploy MUST preserve the full social surface. Both phases target the post-redeploy ProfileRegistry address; `setRelayer(relayerKmsAddress)` is called on the new deployment before linking works. **One coordinated redeploy serves both phases — never two ProfileRegistry redeploys.**
- **D-03: Relayer-as-sender, no paymaster.** `linkTwitter`/`linkFarcaster` are relayer-only; the relayer signs + sends from its GCP-KMS wallet (gas from relayer ETH). Social linking is NOT a user userOp → never consumes the user's 5 sponsored-tx paymaster budget.
- **D-04: Server-side proof verification.** `@privy-io/server-auth` verifies the Twitter OAuth proof; `@farcaster/auth-client` verifies the Farcaster Auth Kit proof. The relayer submits the on-chain link only after successful verification.
- **D-05: proofHash = structured hash of verified claims.** `keccak256` over canonical fields the relayer verified — (user address, platform, handle, OAuth subject/id, issuedAt, attestation nonce) — no PII on-chain.
- **D-06: One active link per handle; reject conflicts.** Relayer keeps an off-chain uniqueness index. A given X/FC handle can be actively linked to only one Call It profile at a time; second account linking the same handle is rejected. Re-linking a *different* handle to your *own* profile is allowed.
- **D-07: Wire badges on ALL AUTH-09 surfaces now.** Reusable `<VerifiedBadge>` in `packages/ui`. Wire it on Profile header, Feed call card, Live Receipt header (via shared `<Receipt>`). Duel parties / Leaderboard rows get **minimal badge-host stubs** sufficient to render + test AUTH-09 now.
- **D-08: Badge data source = subgraph events + relayer enrichment.** Index `SocialLinked`/`SocialUnlinked` (already wired) so feed/list contexts carry verification status in one query; single-profile reads use relayer `/api/profile`. Combined VERIFIED · X / · FC / · X · FC derived from the two linked flags.
- **D-09: Zero mechanical effect (AUTH-10) is a hard invariant + explicit test.** Verification touches no rep / stake-limit / fee / payout path; a test asserts a verified vs unverified user receive identical mechanical treatment.
- **D-10: Build the full feed now.** X API Basic tier (`follows.read` scope) approved (~$100-200/mo, accepted). Farcaster via Neynar's managed Hub API. Both sections ship with collapse toggles.
- **D-11: Feed content rules (AUTH-15).** "From your X" shows ≤10 calls, recency order, includes active calls + active duels, EXCLUDES settled calls; refreshes when feed opens. Matching cross-references the viewer's followed handles against `ProfileRegistry.SocialIdentity.twitterHandle`. "From your Farcaster" mirrors via fid.
- **D-12: Storage = durable Postgres + 1h Redis cache.** Fly Postgres follow-graph table keyed by `privyUserId` + 1-hour Redis cache. **Viewer-only** access. On feed-open: serve cache; on cold/expired: re-fetch + overwrite.
- **D-13: Purge-on-disconnect = Settings action (immediate) + `SocialUnlinked` watcher (backstop).** Settings unlink calls a relayer purge endpoint immediately (deletes Postgres rows + Redis cache); a `SocialUnlinked` event watcher purges as a durable backstop.
- **D-14: Per-platform explicit opt-in (AUTH-16).** "Yes, show me" / "No thanks" on the onboarding follow-graph screen (Screen 3); declined state never renders the section. Consent copy discloses durable, viewer-only, cleared-on-disconnect server-side storage. Opt in later from Settings.

## Summary

Phase 1.5 has **no new Solidity work** — the entire on-chain social surface is already deployed and verified in `packages/contracts/src/ProfileRegistry.sol` (lines 73-128, 168-214) and indexed in the subgraph (`SocialLink` entity + `handleSocialLinked`/`handleSocialUnlinked` mappings, already implemented in `packages/subgraph/src/profile-registry.ts:101-134`). The work is three off-chain layers: (1) a relayer **link-proof verification service** that confirms a Twitter/Farcaster proof and submits `linkTwitter`/`linkFarcaster` from the GCP-KMS wallet; (2) a **VERIFIED badge** primitive + wiring; (3) a **follow-graph data layer** that powers "From your X / Farcaster" feed sections.

The phase splits cleanly into a **buildable-now CORE** with zero paid external dependency, and a **FEED** wave gated on external API access:

- **CORE (no paid dependency, build first):** Twitter link-proof (`@privy-io/server-auth` identity-token read → relayer submit), Farcaster link-proof (`@farcaster/auth-kit` frontend + `@farcaster/auth-client` server verify), `<VerifiedBadge>` + wiring on all AUTH-09 surfaces, link/unlink UI (onboarding + Settings), `setRelayer()` on the post-redeploy ProfileRegistry, AUTH-10 zero-mechanical-effect test, purge-on-disconnect plumbing. **Every dependency here is already installed** (`@privy-io/server-auth@1.32.5`, `@farcaster/auth-kit@0.8.2`) except `@farcaster/auth-client` (one new server dep) and the GCP `oauth-proof` KMS key (the key *type* already exists in `kms-signer.ts:43-48`).
- **FEED (gated, sequence as a later API-key wave):** "From your X" (X API `follows.read`) + "From your Farcaster" (Neynar). Both require external paid/keyed accounts that must be provisioned before this wave can be verified end-to-end. **Critical flag:** the X API pricing model changed on **2026-02-06** — the $200/mo Basic tier is no longer purchasable by *new* developers (legacy subscribers keep it); the new default is consumption-based pay-per-use. The plan must add a `checkpoint:human-verify` to confirm the project's actual X API account status before building against an assumed tier.

**Primary recommendation:** Plan two waves. **Wave A (CORE)** — relayer link/unlink-purge service + `setRelayer` + `<VerifiedBadge>` + link/unlink UI + AUTH-10 parity test, all with already-installed deps. **Wave B (FEED)** — gate behind a `checkpoint:human-verify` that confirms (1) the project's X API account tier/scope and (2) a provisioned Neynar API key, then build the two feed sections + follow-graph Postgres/Redis layer + opt-in consent. CORE has no blocking external dependency and can ship in parallel with Phase 2 immediately.

### Pattern 1: Twitter link-proof (server-side, distinct from session auth)
**What:** The relayer already has a Privy *session* verifier (`verifyAuthToken` → `privyUserId`, in `lib/privy-auth.ts`). The *link proof* is a different operation: read the authenticated user's `linkedAccounts` to extract the **verified** `twitter_oauth.username` + `subject`, then submit on-chain. Privy itself performed the OAuth handshake, so the relayer trusts Privy's attestation rather than re-running OAuth.
**When to use:** AUTH-06 Twitter linking.
**Example:**
```typescript
// Source: docs.privy.io/guide/server/users/get + docs.privy.io/guide/server/authorization/verification
// Reuse the existing getPrivyClient() singleton from lib/privy-auth.ts — do NOT create a second client.
import { getPrivyClient } from '../lib/privy-auth.js';

// 1. Session gate (existing preHandler) confirms request is from privyUserId.
// 2. Read the verified linked accounts (identity token or getUser):
const privy = getPrivyClient();
const user = await privy.getUser(privyUserId);                  // CITED API shape — verify exact signature at build
const twitter = user.linkedAccounts.find((a) => a.type === 'twitter_oauth');
if (!twitter?.username) throw new Error('twitter_not_linked_in_privy');
// 3. twitter.username is Privy-verified; twitter.subject is the stable X user id.
// 4. Map privyUserId → on-chain wallet address (the embedded/connected wallet for this session).
// 5. Uniqueness check (D-06) + proofHash (D-05) + KMS submit (Pattern 4).
```
> **Note:** `getUser` / identity-token method names are CITED, not VERIFIED-by-execution. The `@privy-io/server-auth@1.32.5` API surface must be confirmed at build time (`getUser(did)` vs `getUserById` vs identity-token parse). Flagged in Assumptions Log (A2).

### Pattern 2: Farcaster link-proof (SIWF, server verify)
**What:** Frontend produces a Sign-In-With-Farcaster signed message via `@farcaster/auth-kit`; the relayer verifies it with `@farcaster/auth-client`'s `verifySignInMessage`, checking the app-generated **nonce** and **domain** match the message.
**When to use:** AUTH-07 Farcaster linking.
**Example:**
```typescript
// Source: docs.farcaster.xyz/auth-kit/client/app/verify-sign-in-message
import { createAppClient, viemConnector } from '@farcaster/auth-client';

const appClient = createAppClient({
  relay: 'https://relay.farcaster.xyz',
  ethereum: viemConnector(),
});

const { data, success, fid } = await appClient.verifySignInMessage({
  nonce,                       // MUST equal the nonce your server issued for this attempt
  domain: 'callit.example',    // MUST equal the AuthKitProvider domain
  message,                     // SIWF message from the relay channel
  signature,                   // user's Farcaster wallet signature
});
if (!success) throw new Error('siwf_verification_failed');
```

### Anti-Patterns to Avoid
- **Re-running OAuth in the relayer for Twitter.** Privy already did the handshake — read the verified `linkedAccounts`, don't re-implement Twitter OAuth server-side.
- **Confusing session auth with link proof.** `verifyAuthToken` (session) ≠ reading the verified `twitter_oauth` linked account (proof). The session gate confirms *who*; the linked-account read confirms *what handle they own*.
- **Skipping the nonce/domain check in `verifySignInMessage`.** A SIWF message without server-issued nonce + domain matching is replayable — always pass and validate both.
- **Writing the follow graph on-chain or exposing it to the client.** AUTH-17: server-side only, viewer-only.
- **`display: grid` in any badge that may flow into an OG card.** Satori is flexbox-only (CLAUDE.md). The Tag primitive uses `inline-flex` — keep it.
- **Auto-installing an X API client via `npx --yes` or an unverified npm name.** Use raw `fetch()` with a Bearer token, or verify any package by hand first.
- **Hardcoding the X API tier.** Pricing changed 2026-02-06; verify the account's actual tier before building (Open Question Q1).

## Common Pitfalls

### Pitfall 1: Calling `setRelayer` on the wrong (pre-redeploy) ProfileRegistry
**What goes wrong:** The relayer is authorized on the old ProfileRegistry; the post-redeploy address (shared with Phase 2) still has `relayer == address(0)`, so all links revert `NotRelayer`.
**Why it happens:** Two phases touch one redeploy; the address changes mid-stream.
**How to avoid:** Make `setRelayer(relayerKmsAddress)` an ordered task that runs AFTER the Phase 2 shared redeploy, against the new address, and update `PROFILE_REGISTRY_ARBITRUM_SEPOLIA` in `packages/shared/src/constants/addresses.ts` + subgraph + relayer env in the same wave.
**Warning signs:** `linkTwitter` reverts `NotRelayer` despite a funded KMS wallet.

### Pitfall 2: Treating session auth as proof of handle ownership
**What goes wrong:** Submitting `linkTwitter` using a handle from the request body (client-supplied) instead of the Privy-verified `linkedAccounts` handle → impersonation.
**Why it happens:** The session preHandler returns `privyUserId`, which feels like "verified," but it doesn't verify which Twitter handle the user owns.
**How to avoid:** Always derive the handle from `getUser(privyUserId).linkedAccounts.twitter_oauth.username`, never from the request body.
**Warning signs:** A user can link any handle by editing the request.

### Pitfall 3: Missing nonce/domain validation in SIWF verify
**What goes wrong:** A replayed or cross-app SIWF message links a Farcaster account the requester doesn't control.
**Why it happens:** `verifySignInMessage` accepts nonce + domain as inputs; omitting strict equality checks defeats them.
**How to avoid:** Issue a server-side nonce per attempt (store in Redis with short TTL), require it back, and assert `domain` equals the AuthKitProvider domain.
**Warning signs:** `success: true` for a message your server never issued a nonce for.

### Pitfall 4: X API tier assumption (the big one)
**What goes wrong:** Plan builds against the $200/mo Basic tier; the project's account is on the new pay-per-use model (or has no access), so `follows.read` calls 403/quota-fail in production.
**Why it happens:** X changed the pricing model on **2026-02-06** — Basic is no longer purchasable by new developers; consumption-based pay-per-use is the new default (capped at 2M post reads/mo). `follows.read` per-endpoint pricing is only in the Developer Console, not public pages.
**How to avoid:** Gate the FEED wave behind a `checkpoint:human-verify` confirming the project's actual X API tier, scope (`follows.read`), and `GET /2/users/:id/following` access + quota. Build the section to degrade gracefully to empty/cached (Claude's Discretion) so a quota miss never blocks the main feed.
**Warning signs:** 403 / `client-not-enrolled` on the following endpoint; unexpected per-read charges.

### Pitfall 5: Privy outage breaks linking AND the user can't sign in (Pitfall 16)
**What goes wrong:** Coupling linking to the auth path so a Privy outage cascades.
**Why it happens:** Over-integration.
**How to avoid:** Linking is purely additive and optional (Pitfall 16, ROADMAP). The wallet/SIWE auth path must not depend on the social-link service; a link failure shows a retry, never blocks sign-in or call creation.
**Warning signs:** Sign-in fails when the social-link endpoint is down.

### Pitfall 6: Purge clears Redis but not Postgres (or vice versa)
**What goes wrong:** After unlink, the durable Postgres follow-graph row survives → AUTH-17 privacy violation; or Redis cache survives → stale "From your X" after disconnect.
**Why it happens:** Two stores, one delete.
**How to avoid:** The unlink-purge endpoint deletes BOTH Postgres rows AND Redis keys; the `SocialUnlinked` watcher is the backstop for unlinks done outside the app UI (D-13). Test both stores empty after unlink.
**Warning signs:** Section reappears with stale data after unlink; Postgres rows linger.

### Known Threat Patterns for the relayer link service

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Handle impersonation (link a handle you don't own) | Spoofing | Derive handle from Privy-verified `linkedAccounts` / verified SIWF fid, never from request body (Pitfall 2). |
