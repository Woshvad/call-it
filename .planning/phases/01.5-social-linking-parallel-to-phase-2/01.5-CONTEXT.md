# Phase 1.5: Social linking (parallel to Phase 2) - Context

**Gathered:** 2026-05-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Optional Twitter/Farcaster linking that enriches identity with **zero mechanical effect** on rep, stake limits, fees, or payouts (AUTH-10). Users link via OAuth proof; the relayer verifies server-side and submits the (already-deployed) `linkTwitter`/`linkFarcaster` setters. VERIFIED · X / · FC / · X · FC badges render across surfaces. The Twitter follow-graph powers a "From your X" feed section; Farcaster mirrors as "From your Farcaster". Runs **in parallel with Phase 2** and shares Phase 2's ProfileRegistry redeploy.

**In scope:**
- Relayer social-linking service: verify Twitter OAuth proof (`@privy-io/server-auth`) + Farcaster Auth Kit proof (`@farcaster/auth-client`); submit `linkTwitter`/`linkFarcaster` as `msg.sender == relayer` from the GCP-KMS wallet; off-chain handle-uniqueness index (AUTH-06/07/13).
- Frontend: link/unlink flows (link in onboarding "Connect Socials" + later in Settings; unlink in Settings), Farcaster Auth Kit provider wiring (AUTH-06/07/12).
- `<VerifiedBadge>` primitive + wiring on ALL AUTH-09 surfaces (Profile, Feed card, Live Receipt header via shared `<Receipt>`, + minimal badge-host stubs for Duel/Leaderboard); zero-mechanical-effect test (AUTH-09/10).
- "From your X" + "From your Farcaster" feed sections (≤10 calls, recency, active + active duels, exclude settled, collapse toggles, 1h cache) — X API Basic + Neynar (AUTH-14/15/18).
- Follow-graph data layer: durable Fly Postgres store (viewer-scoped) + 1h Redis cache; purge-on-disconnect (AUTH-17).
- Onboarding per-platform opt-in screen (Yes/No), declined never shown, consent discloses durable storage (AUTH-16).
- Subgraph `SocialLinked`/`SocialUnlinked` indexing (coordinated with Phase 2's subgraph extension).

**Out of scope:**
- New ProfileRegistry contract code — the social surface (`linkTwitter`/`linkFarcaster`/`unlink*`/`SocialIdentity`/`relayer`/`setRelayer`/events) is ALREADY deployed. This phase shares Phase 2's single ProfileRegistry redeploy.
- Full Duel page (→ Phase 3) + full Leaderboard page (→ Phase 7) — Phase 1.5 only stubs the badge hosts.
- Settled Receipt badge wiring (→ Phase 4 when the Settled Receipt is built; primitive is ready).
- Any rep/stake/fee/payout interaction (forbidden by AUTH-10).
- Notifications of new "From your X" calls (Phase 2 owns the generic notification center; could extend later).

<decisions>
## Implementation Decisions

### Contract surface (already deployed — coordination only)
- **D-01: No new ProfileRegistry contract code.** The deployed ProfileRegistry already has `linkTwitter`/`linkFarcaster` (relayer-only), `unlinkTwitter`/`unlinkFarcaster` (user self-service), `SocialIdentity` storage (twitter/farcaster handle + proofHash + linkedAt; 50-byte handle cap per AUTH-13), `relayer` + `setRelayer`, and `SocialLinked`/`SocialUnlinked` events. Phase 1.5 wires the relayer (`setRelayer`) and builds the service/frontend/data layers on top.
- **D-02: Shared ProfileRegistry redeploy with Phase 2.** Phase 2 (02-CONTEXT D-04) redeploys ProfileRegistry to add the `authorizedRepWriters` set. That redeploy MUST preserve the full social surface. Both phases target the post-redeploy ProfileRegistry address; `setRelayer(relayerKmsAddress)` is called on the new deployment before linking works. **One coordinated redeploy serves both phases — never two ProfileRegistry redeploys.**

### Linking flow (relayer + proof)
- **D-03: Relayer-as-sender, no paymaster.** `linkTwitter`/`linkFarcaster` are relayer-only (`msg.sender == relayer`); the relayer signs + sends from its GCP-KMS wallet (gas from relayer ETH). Social linking is NOT a user userOp → it never consumes the user's 5 sponsored-tx paymaster budget.
- **D-04: Server-side proof verification.** `@privy-io/server-auth` verifies the Twitter OAuth proof; `@farcaster/auth-client` verifies the Farcaster Auth Kit proof. The relayer submits the on-chain link only after successful verification.
- **D-05: proofHash = structured hash of verified claims.** `keccak256` over canonical fields the relayer verified — (user address, platform, handle, OAuth subject/id, issuedAt, attestation nonce) — no PII on-chain. Provides an auditable (user↔handle) binding commitment; mirrors the project's EIP-712/KMS-attestation pattern (Phase 4).
- **D-06: One active link per handle; reject conflicts.** The relayer keeps an off-chain uniqueness index. A given X/FC handle can be actively linked to only one Call It profile at a time; a second account linking the same handle is rejected ("already linked to another profile"). Re-linking a *different* handle to your *own* profile is allowed (last valid proof wins for your account). Anti-impersonation guard.

### VERIFIED badges
- **D-07: Wire badges on ALL AUTH-09 surfaces now.** Build a reusable `<VerifiedBadge>` primitive in `packages/ui`. Wire it on Profile header, Feed call card, and the Live Receipt header (via shared `<Receipt>` — couples cleanly with parallel Phase 2). For not-yet-built surfaces — Duel parties (Phase 3), Leaderboard rows (Phase 7) — build **minimal badge-host stubs** sufficient to render + test AUTH-09 now; full page build remains Phase 3/7.
- **D-08: Badge data source = subgraph events + relayer enrichment.** Index `SocialLinked`/`SocialUnlinked` in the subgraph so feed/list contexts carry verification status in one query (reflects unlink); single-profile reads use the relayer `/api/profile` endpoint (D-13 pattern) to enrich. Combined VERIFIED · X / · FC / · X · FC derived from the two linked flags.
- **D-09: Zero mechanical effect (AUTH-10) is a hard invariant + explicit test.** Verification touches no rep / stake-limit / fee / payout path; a test asserts a verified vs unverified user receive identical mechanical treatment (mirrors Phase 1's parity-gate discipline).

### Follow-graph feed ("From your X / Farcaster")
- **D-10: Build the full feed now.** X API **Basic tier** (`follows.read` scope) approved for the Twitter follow-graph — this resolves the STATE.md X-API budget blocker for this item (~$100-200/mo, accepted). Farcaster follow-graph via **Neynar's managed Hub API** (handle/fid resolution + follows). Both sections ship with collapse toggles (AUTH-18).
- **D-11: Feed content rules (AUTH-15).** "From your X" shows ≤10 calls, recency order, includes active calls + active duels, EXCLUDES settled calls; refreshes when the feed opens. Matching cross-references the viewer's followed handles against `ProfileRegistry.SocialIdentity.twitterHandle` (linked Call It users). "From your Farcaster" mirrors via fid.
- **D-12: Storage = durable Postgres + 1h Redis cache.** A Fly Postgres follow-graph table keyed by the authenticated viewer's `privyUserId` (followed handles/fids), plus a 1-hour Redis cache (AUTH-15). **Viewer-only** access — the relayer never exposes another user's graph. On feed-open: serve cache; on cold/expired: re-fetch from X/Neynar + overwrite.
- **D-13: Purge-on-disconnect = Settings action (immediate) + `SocialUnlinked` watcher (backstop).** The Settings unlink flow calls a relayer purge endpoint immediately (deletes Postgres rows + Redis cache); a `SocialUnlinked` event watcher purges as a durable backstop for unlinks that happen outside the app UI. Belt-and-suspenders for the AUTH-17 privacy guarantee.

### Onboarding opt-in
- **D-14: Per-platform explicit opt-in (AUTH-16).** "Yes, show me" / "No thanks" on the existing onboarding follow-graph screen (Phase 1 Screen 3); declined state never renders the section (never bypassed). The consent copy discloses that the follow graph is stored server-side (durable, viewer-only, cleared on disconnect) per D-12/D-13. Users can also opt in later from Settings.

### Claude's Discretion
- Handle normalization (case-folding, `@`-prefix stripping) for matching.
- X API rate-limit / quota degradation (serve cached or empty section gracefully; never block the main feed).
- Subgraph schema additions for `SocialLinked`/`SocialUnlinked` (Phase 0 schema may already include; extend + redeploy alongside Phase 2).
- `<VerifiedBadge>` visual treatment within the neobrutalist system (reuse the Tag primitive).
- Relayer endpoint shapes (`/api/social/link`, unlink-purge, `/api/feed/from-your-x`, `/api/feed/from-your-farcaster`).
- Attestation-nonce management for proofHash.

</decisions>

<specifics>
## Specific Ideas

- **Phase 1.5 runs PARALLEL to Phase 2 and shares the ProfileRegistry redeploy** — coordinate as ONE redeploy that preserves the social surface AND adds `authorizedRepWriters`. Two ProfileRegistry redeploys would be a planning error.
- **Build the `<VerifiedBadge>` primitive + shared `<Receipt>` integration early** — it directly feeds Phase 2's Live Receipt (parallel).
- **Durable follow-graph storage (D-12) raises the privacy bar** — purge-on-unlink (D-13) must clear Postgres AND Redis, and the opt-in consent (D-14) must explicitly disclose durable server-side storage.
- **Linking is independent of auth (Pitfall 16)** — a Privy outage doesn't break the wallet path; social linking is purely additive and optional.
- **Zero mechanical effect (AUTH-10) is a hard, tested invariant** — verification must not touch any rep/stake/fee/payout path.

</specifics>
