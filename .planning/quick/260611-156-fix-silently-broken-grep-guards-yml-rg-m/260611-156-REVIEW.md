## Summary

Adversarial review of the grep-guards resurrection focused on semantic weakening, lost
coverage, and future silent-failure modes — the things the green CI runs cannot prove.
All claims below were verified empirically against rg 14.1.1 and the actual repo, not
taken from the commit message.

**Verdict on the six critical questions:**

1. **`0xff970a61[0-9a-f]` catches every real paste — PROVEN.** The full address is
   `0x` + 40 contiguous hex chars; any contiguous paste of ≥11 chars has a hex char at
   position 11, so it trips (case covered by `--ignore-case`, which makes `[0-9a-f]`
   match `A-F`). A ≤10-char fragment is not a usable address. Empirically confirmed:
   `startsWith("0xff970a61")` and the ellipsis form `0xFF970A61...DB5CC8` pass;
   `0xff970a61a0` trips. The only constructible evasion the old pattern caught and the
   new one misses is a deliberately split literal (`'0xff970a61' + 'a04b…'`) — an
   adversarial pattern grep guards never claimed to stop. See IN-01 for the scoping
   caveat (the narrowing is global, not file-scoped).
2. **Glob rewrite loses `.cts`/`.mts` (and `.vue`)** vs. what rg 14's `--type ts`/
   `--type js` would have selected (`ts: *.cts, *.mts, *.ts, *.tsx`; `js: *.cjs, *.js,
   *.jsx, *.mjs, *.vue`). Zero such files exist in the repo today, and phase-1-gates'
   unrestricted `grep -r` backstops full-address pastes in any file type. `.d.ts` IS
   covered (`**/*.ts` matches it). `.rs` is genuinely covered now (positive glob, no
   type-AND). See IN-02.
3. **rg negative globs are exact-file — PROVEN.** A glob containing `/` is
   root-anchored in rg's gitignore-style matching: a planted copy at
   `nested/packages/shared/src/constants/usdc.ts` still trips while the root-path file
   is excluded; `usdc.ts.bak` is never selected by `**/*.ts` in the first place.
   **However, phase-1-gates' `grep -vF` allowlist filters are NOT exact-file** —
   proven masking bypass, see WR-02.
4. **IN-03 promotion cannot false-positive today** (`apps/web/lib/privy-config.ts:56`
   has `NEXT_PUBLIC_NETWORK` and `mainnet` on one line). A refactor that moves the file
   outside the 3 hard-coded dirs or splits the ternary across lines bricks the gate
   **fail-loud** — the correct trade. The real residual risk is the opposite: the gate
   is vacuously satisfiable by a comment line. See IN-03.
5. **No new silent-failure path introduced.** Install step is fail-loud (`&&` chain,
   no `|| true`); `command -v rg` gates cover exit 127. But a residual class survives:
   rg exit 2 (bad glob/regex/IO error) still reads as "no match" → PASS. See WR-03.
6. **`command -v rg` gates present in all FOUR rg-using run blocks — CONFIRMED**
   (usdc-paste ts/js check, usdc-paste sol check, usdc-paste allowlist check,
   solidity-pragma check).

The pragma pin in SettleTrace.t.sol is correct and is the last floating pragma in
tracked sources (verified: `git grep 'pragma solidity'` shows no non-`=0.8.30` pragma
outside `packages/contracts/lib/`).

No Critical findings. No weakening of guard semantics against any real address paste.

## Warnings

### WR-01: scripts/phase-0-smoke.ts still embeds the OLD vacuous guard — the fixed bug class survives in its mirror

**File:** `scripts/phase-0-smoke.ts:118`
**Issue:** The smoke script's USDC.e check embeds the exact pre-fix command:
`--type ts --type js --glob "**/*.rs"` (the vacuous AND-conjunction that selects ZERO
files), the bare `0xff970a61` pattern, only ONE fixture exclusion, and a trailing
`|| true` that swallows exit 127 if rg is missing. This is the precise
green-but-vacuous defect this commit fixed in CI, left alive in the mandatory
post-deploy smoke script (§19.11). The commit even names this embedded string as an
allowlisted guard self-reference (grep-guards.yml comment) — acknowledging its
existence while leaving it broken. Worse, it is doubly divergent: if someone later
"fixes" only the file filter without also updating the pattern and allowlist, the smoke
check will false-positive on `packages/shared/test/usdc.test.ts` and the
self-references. The pragma mirror at line 124 shares the `|| true`-swallows-everything
weakness.
**Fix:** Sync `step2GrepGuards` with the resurrected CI commands: pure positive globs
(ts/tsx/js/jsx/mjs/cjs/rs), pattern `0xff970a61[0-9a-f]`, both fixture exclusions
(`!packages/shared/src/constants/usdc.ts`, `!packages/shared/test/usdc.test.ts`), and a
`command -v rg` precheck before the loop so a missing local rg fails the smoke run
instead of passing it. Note the embedded string must keep a bare `0xff970a61` *inside a
non-matching context* or itself be added to the guard's documented self-references —
after this sync, the new command string will contain `0xff970a61[0-9a-f]`, whose
literal text does not match the regex (the char after `61` is `[`), so it stays clean.

### WR-02: phase-1-gates allowlist filters are line-substring matches, not exact-file matches — proven masking bypass

**File:** `.github/workflows/phase-1-gates.yml:161-164` (and duplicate at `:501-504`)
**Issue:** `grep -vF "packages/shared/src/constants/usdc.ts"` drops any output LINE
containing that substring — in the path field OR the content field. Empirically proven:
a line `const BAD = "0xFF970A61A04b...CC8"; // copied from
packages/shared/src/constants/usdc.ts` in a non-fixture file is filtered out → real
violation masked. Likewise, a file named `packages/shared/src/constants/usdc.ts.bak`
(in scope, since this grep has no file-type restriction) would be masked because its
path contains the fixture path as a prefix. The `-v` → `-vF` change in this commit
strictly *strengthened* the filter (regex `.` no longer wildcards), but did not anchor
it. grep-guards' rg negative globs do not share this flaw (root-anchored exact-file),
so defense-in-depth currently holds for `.ts`/`.sol` — but phase-1-gates is the ONLY
guard covering other file types (.json, .env, .md, .sh), and there the mask is the only
allowlist.
**Fix:** Anchor each filter to the path field of grep's `path:line:content` output
(grep runs against `.`, so paths are `./`-prefixed):
```bash
| grep -v '^\./packages/shared/src/constants/usdc\.ts:' \
| grep -v '^\./packages/shared/test/usdc\.test\.ts:' \
| grep -v '^\./packages/contracts/src/constants/USDC\.sol:' \
| grep -v '^\./packages/contracts/test/USDC\.t\.sol:' || true)
```

### WR-03: `command -v rg` gates cover only exit 127 — rg exit 2 still reads as green

**File:** `.github/workflows/grep-guards.yml:67-81, 96-105, 126-136, 168-176`
**Issue:** The new gates close the missing-binary hole, but the failure mode that made
this job vacuous has a sibling: ripgrep exits 2 on usage/runtime errors (invalid glob,
bad regex, IO error). In the `if rg ...; then` blocks, exit 2 takes the else-branch →
"PASS" is printed. In the `VIOLATIONS=$(rg ... | ... || true)` blocks (allowlist,
pragma), exit 2 is swallowed by `|| true` → empty VIOLATIONS → PASS. A future edit that
typos a glob re-creates green-but-vacuous with rg fully installed, and `command -v`
won't catch it. (GitHub's default `bash -e` shell has no pipefail, so the piped forms
can't catch it either.)
**Fix:** Discriminate exit codes — rg's contract is 0=match, 1=no-match, ≥2=error:
```bash
set +e
rg <args> .
status=$?
set -e
if [ "$status" -eq 0 ]; then echo "::error::..."; exit 1; fi
if [ "$status" -ne 1 ]; then echo "::error::rg failed (exit $status) — guard did not run"; exit 1; fi
```

## Info

### IN-01: Continuation-char narrowing is pattern-global, not scoped to the two known self-references

**File:** `.github/workflows/grep-guards.yml:80, 105`
**Issue:** The stated motivation is two specific self-references
(`scripts/phase-0-smoke.ts:118`, `apps/web/tests/onboarding.spec.ts:134`), but the
relaxation applies everywhere: any future bare-prefix usage — a `startsWith('0xff970a61')`
runtime guard, a truncated display string, a split-literal construction — passes
silently in any file. Since a ≤10-char fragment cannot transact and prefix-comparison
code is virtually always negative-assertion style, this is an accepted trade (and
mirrors phase-1-gates' existing `0xFF970A61[A-Fa-f0-9]+`), but it is a one-way door:
nothing now alerts on new bare-prefix references accumulating.
**Fix:** None required for v1. If tightened later, keep the bare-prefix pattern and
file-scope the two self-references with negative globs instead.

### IN-02: Glob rewrite drops `.cts`/`.mts` relative to the intended `--type ts` coverage

**File:** `.github/workflows/grep-guards.yml:68-74`
**Issue:** rg 14's `ts` type is `*.cts, *.mts, *.ts, *.tsx`; `js` includes `*.vue`. The
pure-glob rewrite omits `.cts`, `.mts`, `.vue`. Zero such files exist in the repo
today, and phase-1-gates' file-type-unrestricted grep catches full-address pastes in
them, so there is no live gap — but a future `.mts` config script would silently fall
outside this guard.
**Fix:** Add `--glob '**/*.cts' --glob '**/*.mts'` for parity (skip `.vue` — stack is
React).

### IN-03: NEXT_PUBLIC_NETWORK gate asserts string presence, not behavior — vacuously satisfiable by a comment

**File:** `.github/workflows/phase-1-gates.yml:524-535`
**Issue:** The promotion to `exit 1` is correct and the false-positive risk is the good
kind (fail-loud brick if the guard moves out of `apps/web/{src,app,lib}` or the ternary
splits across lines — `apps/web/src/` doesn't even exist today; its grep error is
harmlessly suppressed). The residual risk runs the other way: `privy-config.ts:37` is a
*comment* containing both `NEXT_PUBLIC_NETWORK` and `mainnet`, so deleting the actual
network branching at line 56 while keeping the docblock still passes the gate. Also,
`grep -q "testnet\|mainnet"` uses GNU-BRE alternation — fine on ubuntu runners, but
`grep -qE 'testnet|mainnet'` is the portable spelling.
**Fix:** Match the executable line shape, e.g.
`grep -rE "NEXT_PUBLIC_NETWORK.*===.*'(mainnet|testnet)'" apps/web/app/ apps/web/lib/`
and drop the nonexistent `apps/web/src/`.

### IN-04: Solidity USDC.e check has fewer dir exclusions than the ts/js check

**File:** `.github/workflows/grep-guards.yml:97-102`
**Issue:** The `.sol` check excludes only `node_modules/out/target`, while the ts/js
check also excludes `.next/.turbo/dist`. With `--hidden --no-ignore`, a `.sol` artifact
under `dist/` would trip the guard. This fails loud (noise), not silent, and `.sol`
files don't land in those dirs in practice — cosmetic asymmetry only. (It also does not
exclude `**/lib/**`; `packages/contracts/lib/` is untracked and absent in CI checkouts,
and OZ sources contain no USDC.e address, so no live effect.)
**Fix:** Mirror the exclusion set across both steps for consistency.
