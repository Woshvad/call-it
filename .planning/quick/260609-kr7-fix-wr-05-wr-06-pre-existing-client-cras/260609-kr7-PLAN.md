<task type="auto">
  <name>Task 2: WR-06 — FileReader-based evidence encode + max-size guard</name>
  <files>apps/web/app/call/[id]/page.tsx</files>
  <action>
The DisputeModal evidence encode (currently line 526) does `btoa(String.fromCharCode(...new Uint8Array(content)))`, spreading the whole file's byte array into function arguments — this throws `RangeError: Maximum call stack size exceeded` for files of a few hundred KB+, aborting the $5-bond dispute-evidence upload (SETTLE-26). Per WR-06, fix the encode AND add a size guard:

1. In `handleEvidenceUpload` (currently lines 522-541), replace the `file.arrayBuffer()` + `btoa(String.fromCharCode(...))` lines with a `FileReader`-based encode: wrap a `new Promise<string>` that creates a `FileReader`, sets `onload` to resolve with `(reader.result as string).split(',')[1]` (the base64 portion after the `data:` prefix), sets `onerror` to reject, and calls `reader.readAsDataURL(file)`. Await that promise to get `base64`, then POST it to the existing `${relayerUrl}/api/disputes/evidence` endpoint exactly as before (same body shape: `{ content: base64, filename: file.name, mimeType: file.type }`). Keep the existing try/catch/finally, `setUploadingEvidence`, `setEvidenceCid`, and `setEvidenceHash` handling unchanged.

2. Add a max-size guard before upload on the `<input type="file">` onChange (currently lines 636-643). Define a constant `MAX_EVIDENCE_BYTES = 5 * 1024 * 1024` (5 MB) near the handler. In the `onChange`, after grabbing `file`, if `file.size > MAX_EVIDENCE_BYTES` set the existing `toast` to an error (e.g. "Evidence file too large — max 5 MB") via `setToast({ text: ..., isError: true })`, clear/skip setting `evidenceFile`, and return early WITHOUT calling `handleEvidenceUpload`. Only proceed to `setEvidenceFile(file)` + `void handleEvidenceUpload(file)` when within the limit.

Do NOT touch the approve/raiseDispute handlers, the modal layout, the `isMobile` clamp, or any other DisputeModal logic.
  </action>

</task>
