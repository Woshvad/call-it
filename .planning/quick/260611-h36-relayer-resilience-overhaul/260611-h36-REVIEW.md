## Warnings

### WR-01: Call marked terminal on tx submission, not confirmation â€” a mined-but-reverted settle is silently abandoned until restart

**File:** `apps/relayer/src/workers/settlement-poller.ts:294-309`
**Issue:** `settlePythCall` returns the hash from `walletClient.writeContract` (broadcast, no receipt wait â€” pyth-adapter.ts:383). The poller then sets `info.terminal = true` and clears counters. viem's implicit gas estimation catches most reverts pre-broadcast, but a race (e.g. an external settle or price-update landing between estimation and inclusion) yields a mined-reverted tx â†’ the call is flagged terminal, skipped by every future tick, with no alert â€” the exact "auto-settlement structurally dead" failure this poller exists to fix, now silent. State resets on restart, which masks rather than fixes it.
**Fix:** After `settlePythCall`, `await publicClient.waitForTransactionReceipt({ hash: txHash })` and only set `info.terminal = true` when `receipt.status === 'success'`; on `'reverted'`, route through `recordFailure(idStr, 'tx_reverted')`. Alternatively re-read the call and only mark terminal when on-chain status is Settled/Disputed.

### WR-02: `err.message` logging can leak the signer account address, contradicting the module's own T-h36-01 claim

**File:** `apps/relayer/src/workers/settlement-poller.ts:276,311,367` (claim at lines 30-31)
**Issue:** viem `ContractFunctionExecutionError.message` embeds the full "Request Arguments" block including `from: 0x<signer address>`. `recordFailure` and the candidate-error handler log `err.message` verbatim, so any failed settle writes the signer address into pino â€” while the header asserts "this module NEVER logs the signer account address". No key material leaks (verified: privateKeyToAccount errors are swallowed sanitized; the key never reaches a log object), but the documented invariant is false on the first revert.
**Fix:** Truncate to the error's short name/first line, e.g. `reason: err instanceof BaseError ? err.shortMessage : String(err).split('\n')[0]`, or strip `from:`/`Request Arguments` lines before logging. Or amend the header to claim only key-material safety.

### WR-03: Stale challenge-id snapshots are served silently as fresh, and the GLOBAL breaker couples feed failures to settlement inputs

**File:** `apps/relayer/src/lib/subgraph-client.ts:702-710` (breaker: 380-425)
**Issue:** Two compounding effects. (a) `queryAcceptedChallengeIds` goes through `executeQuery`, which drops the `stale` flag â€” a â‰¤1h-old snapshot is returned as if live, so a duel accepted after the snapshot is omitted from `settle(...)` with no log (unlike the thrownâ†’`[]` path, which warns). (b) `breakerOpenUntil` is a single global: any feed/profile 429 within the 120s cooldown short-circuits the (low-volume, settlement-critical) challenge query to stale-or-throw even when Studio would have served it. Contract validates the ids it receives but cannot add missing ones â€” accepted duels for that call go unsettled in the settle tx.
**Fix:** In `queryAcceptedChallengeIds`, call `executeQueryWithMeta` and on `stale: true` log a warn with the snapshot age (mirroring the `[]`-fallback warn); optionally bypass the breaker (direct `executeQueryUpstream` with one retry) for this query since it runs at most once per candidate per tick.

### WR-04: Breaker half-open has no single-probe gate, and opens on deterministic non-infra errors

**File:** `apps/relayer/src/lib/subgraph-client.ts:398-424`
**Issue:** (a) When `breakerOpenUntil` lapses, EVERY in-flight caller passes the `Date.now()` check and probes upstream concurrently â€” a thundering herd at exactly the moment the rate-limited Studio endpoint is recovering (the 429-storm being fixed). The herd is bounded in practice by the routes' L1 caches, but nothing enforces "single probe". (b) The breaker opens on ANY throw, including deterministic GraphQL validation errors and `SUBGRAPH_STUDIO_URL is not configured` â€” a schema typo in one query masks itself as a 120s outage and serves up-to-1h-stale data for all queries.
**Fix:** (a) Add an in-flight probe promise: while a half-open probe is pending, concurrent callers await it (or serve stale). (b) Don't open the breaker for `GraphQL errors:` / missing-config throws â€” rethrow those directly so they surface as bugs, not outages.

### WR-05: Synthetic-alert nonce replay window after Redis recovery (memory fallback only consulted when SETNX throws)

**File:** `apps/relayer/src/workers/synthetic-event-handler.ts:106-135`
**Issue:** A nonce claimed via the in-memory fallback during a Redis outage never reaches Redis. When Redis recovers, replaying the same captured HMAC'd request hits the happy path: `redis.set(..., 'NX')` succeeds (Redis never saw the nonce) â†’ accepted â†’ duplicate alert fired. The memory cache still holds the nonce but is never checked when Redis is healthy. Bounded by the 5-minute timestamp window and HMAC possession, so impact is alert spam, not auth bypass â€” but the replay guard the fallback claims to "preserve" has a hole exactly at the outage/recovery boundary.
**Fix:** Check `nonceFallbackCache.get(nonceKey)` BEFORE the Redis SETNX (reject if present), and also write to the memory cache on the successful Redis path so both stores stay superset-consistent on a single machine.

### WR-06: Discovery has no detection for a silently-empty frontier â€” wrong registry address or ABI drift looks identical to "no calls exist"

**File:** `apps/relayer/src/workers/settlement-poller.ts:319-344`
**Issue:** The zeroed-tuple stop is the only discovery signal. A misconfigured `callRegistryAddress`, a redeployed registry (this repo has redeployed CR before â€” 05.1), or ABI drift that decodes `caller` as zero halts the frontier at id 1 forever: the poller logs `settlement_poller_started`, ticks happily, finds zero candidates, and never alerts â€” an unattended settlement worker that is structurally blind while appearing healthy. (Mid-range gaps can't occur with the current monotonic registry, so the documented stop heuristic is otherwise sound and does not infinite-loop.)
**Fix:** Cheap invariant probe: if `frontier === 1n` (or `known.size === 0`) persists for N ticks while the registry's `nextCallId()`/`callCount()` (or a known-good sentinel id) reports calls exist, log error + fire the infra P1 once. Even just exposing `frontier`/`known` via the existing `/health` payload would let the soak monitor catch it.
