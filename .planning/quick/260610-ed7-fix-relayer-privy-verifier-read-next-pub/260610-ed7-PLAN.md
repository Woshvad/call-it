<objective>
Fix the relayer Privy session verifier so it builds the PrivyClient with the correct
(frontend-matching) app id. Currently getPrivyClient() reads a stale `PRIVY_APP_ID`
env var, so `verifyAuthToken` rejects every valid frontend token with
`401 { code: 'invalid_session' }` → sign-in bounces back to `/signin`.

The relayer's correct app id is loaded from GCP as `NEXT_PUBLIC_PRIVY_APP_ID`
(secret-manager.ts mirrors it into process.env at boot) and matches the frontend
app `cmphfvdy500230el7nv2vfo9x`. Change the app-id source to prefer
`NEXT_PUBLIC_PRIVY_APP_ID`, keeping `PRIVY_APP_ID` as a legacy fallback so existing
tests keep passing.

Purpose: Unblock all user-facing relayer session auth (onboarding state, calls, etc.)
so live Sepolia sign-in stops bouncing to /signin.
Output: One-line app-id resolution change + updated doc comment + updated throw message
in apps/relayer/src/lib/privy-auth.ts. No other files touched.
</objective>
