# Quick 260611-m72: Settings Save Handle Honesty Summary

**One-liner:** Settings Save Handle rewired to the proven call-page write pattern — ensureActiveChain + chainId-pinned writeContractAsync + mined-receipt-gated success + honest error taxonomy (was sync fire-and-forget whose success line meant "submitted", not "mined").

## What Changed

Single file: `apps/web/app/profile/[address]/settings/page.tsx`

1. **Chain guard + pin (T-m72-02).** `handleSetDisplayHandle` is now async: `await ensureActiveChain()` before the write, `chainId: ACTIVE_CHAIN_ID` on `writeContractAsync`, and `chainId: ACTIVE_CHAIN_ID as ActiveChainId` on `waitForTransactionReceipt`. This was the one write site missed by the 17-site chain sweep.
2. **Receipt-gated success (T-m72-01).** Hook `isSuccess` (`handleWriteSuccess`) replaced with local `handleSaved`, set ONLY when `receipt.status === 'success'`. A non-success receipt sets `'Save failed on-chain — try again.'`. "Handle saved on-chain." is now true.
3. **Honest error taxonomy (T-m72-03).** Catch block in priority order:
   - `isUserRejection(err)` (reused from `app/new/lib/call-created-log.ts`) → "Transaction rejected — your username was not saved."
   - error chain contains "insufficient funds" (case-insensitive; checked via `err.walk().message` — the live failure mode, embedded wallet has 0 ETH) → faucet/funded-wallet copy
   - otherwise → viem `BaseError.shortMessage` (trimmed) or first line of `Error.message`, fallback "Save failed — try again." `ensureActiveChain`'s wrong-network message surfaces here as-is.
   All paths land in the existing `handleError` inline render near the input.
4. **Full-flow busy state.** Hook `isPending` (`isWritingHandle`) replaced with local `isSavingHandle` spanning chain-switch → submit → mined receipt; drives disabled/cursor/opacity/label.
5. **Header doc** updated to document the new sequence and the quick-fix provenance.
